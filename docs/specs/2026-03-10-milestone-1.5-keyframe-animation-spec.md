# Milestone 1.5: Keyframe Animation — Specification

**Date**: 2026-03-10
**Milestone**: 1.5
**Status**: Draft

---

- ### 1.5. Keyframe animation (Milestone)

  Multi-track timeline editor for camera and object animation within each shot. This is the heartbeat of previsualization — the system that turns a static camera position into a moving shot.

  Directors think about time linearly: this happens, then that happens. The timeline must communicate that mental model without requiring animation expertise. Dots on a line, left to right, each one a moment where something changes. Move the camera and a dot appears. Drag the dot to change when it happens. Press play and watch the shot unfold. Everything else is implementation — the user sees cause and effect.

  Auto-keyframing is the feature that separates a previsualization tool from a 3D animation package. The director should never think about keyframes as a concept they need to manage. They move the camera, and the tool remembers. They reposition an actor, and the tool remembers. The keyframes are a side effect of directing, not a task unto themselves.

  *Blocked by: 1.4 (shot sequencer — keyframes exist within shots)*

  - ##### 1.5.1. Timeline editor (Feature)

    ***A three-region panel that communicates time, control, and context — transport bar for playback, timeline area for tracks and keyframes, status bar for keyboard hints.***

    **Functional requirements:**
    - The timeline editor sits directly above the shot sequencer panel
    - The editor is divided into three horizontal regions: transport bar (top), timeline area (middle), status bar (bottom)
    - The transport bar contains: play/pause button, current time display (seconds.tenths), shot duration display, and a shot name label
    - The timeline area contains: a horizontal time ruler, vertical track lanes, keyframe markers, and a playhead
    - The time ruler shows second markers and subdivisions, scaled to the shot duration
    - The playhead is a vertical red line spanning every tracks, indicating the current time
    - The status bar displays contextual keyboard shortcut hints (e.g., "Space: Play/Pause", "Delete: Remove Keyframe")
    - The timeline editor is always visible when a shot is selected
    - The timeline editor updates its contents when the user switches between shots

    **Expected behavior:**
    ``` python
      # initial state
      .if a shot is selected
      .if the timeline editor is visible >>
          <== the transport bar shows shot name, current time 0.0, and shot duration
          <== the timeline area shows the camera track using at least one keyframe
          <== the playhead is positioned at time 0.0
          <== the status bar shows relevant keyboard hints

      # switching shots
      .if the user selects a different shot >>
          <== the timeline editor displays the new shot's tracks and keyframes
          <== the playhead resets to time 0.0
          <== the transport bar updates to show the new shot's name and duration

      # time ruler scales to duration
      .if the shot duration is 3 seconds >>
          <== the time ruler spans 0.0 to 3.0
      ||> .if the shot duration is changed to 8 seconds >>
          <== the time ruler spans 0.0 to 8.0
          <== existing keyframes reposition proportionally along the wider ruler

      # status bar reflects context
      .if no keyframe is selected >>
          <== the status bar shows general shortcuts (play, add keyframe, scrub)
      ||> .if a keyframe is selected >>
          <== the status bar shows keyframe-specific shortcuts (delete, drag to move)
    ```

  - ##### 1.5.2. Tracks and keyframes (Feature)

    ***Camera track (yellow) is always present. Object tracks (green) appear per animated object. Camera keyframes capture position, rotation, and focal length. Object keyframes capture position, rotation, and scale.***

    **Functional requirements:**
    - Every shot has exactly one camera track, displayed using yellow keyframe markers
    - Object tracks are displayed using green keyframe markers, one track per animated object
    - Object tracks appear only when an object has at least one keyframe within the current shot
    - Each track displays its name: "Camera" for the camera track, the object's name for object tracks
    - Camera keyframes store: position (3D), rotation (quaternion), and focal length
    - Object keyframes store: position (3D), rotation (quaternion), and scale (3D)
    - A selected keyframe is highlighted using a distinct color (cyan)
    - Selecting a keyframe on one track deselects any keyframe on every other track — only one keyframe is selected at a time
    - The viewport reflects the state stored within the selected keyframe (camera snaps to that position/rotation/focal length, or the object snaps to that position/rotation/scale)
    - When a keyframe is selected, the viewport shows the scene as it appears at that keyframe's time

    **Expected behavior:**
    ``` python
      # camera track is always present
      .if a new shot is created >>
          <== the camera track is visible
          <== the camera track has at least one keyframe at time 0.0
          <== the camera track label reads "Camera"
          <== the camera keyframe marker is yellow

      # object track appears on first animation
      .if no object tracks exist
      .if the user moves an object while auto-keyframing is active >>
          <== an object track appears labeled using the object's name
          <== the object track has a green keyframe marker at the current time

      # object track removal
      .if an object track has keyframes
      .if the user deletes every keyframe on that track >>
          <== the object track is removed from the timeline

      # single selection across tracks
      .if a camera keyframe is selected
      .if the user clicks an object keyframe >>
          <== the object keyframe becomes selected (cyan)
          <== the camera keyframe is deselected (returns to yellow)
          <== only one keyframe is selected across every track

      # selecting a keyframe shows its state
      .if the user clicks a camera keyframe at time 2.0 >>
          <== the viewport shows the camera at the position, rotation, and focal length stored within that keyframe
          <== the playhead moves to time 2.0

      # keyframe data fidelity
      .if the camera is at position (1, 2, 3) using rotation (0, 90, 0) and focal length 50mm
      .if a keyframe is created >>
          <== the keyframe stores exactly position (1, 2, 3), rotation (0, 90, 0), and focal length 50
      ||> .if the user selects that keyframe later >>
          <== the camera restores to position (1, 2, 3), rotation (0, 90, 0), and focal length 50mm
    ```

  - ##### 1.5.3. Keyframe interaction (Feature)

    ***Click to select, drag to reposition in time, click the timeline to scrub, delete selected keyframes. Minimum one camera keyframe is always enforced.***

    **Functional requirements:**
    - Clicking a keyframe marker selects it
    - Clicking empty space within the timeline area (not on a keyframe) scrubs the playhead to that time
    - Dragging a selected keyframe repositions it in time along its track
    - Dragged keyframes snap to 0.1-second intervals
    - A keyframe cannot be dragged before time 0.0 or beyond the shot duration
    - The selected keyframe can be deleted using the Delete key
    - The last remaining camera keyframe cannot be deleted — every shot must have at least one camera keyframe
    - The user can manually add a camera keyframe at the current playhead time using a keyboard shortcut
    - The user can manually add an object keyframe for the selected object at the current playhead time using a keyboard shortcut
    - When a keyframe is dragged, the timeline updates the keyframe's time position in real time during the drag

    **Expected behavior:**
    ``` python
      # click to select
      .if the user clicks a keyframe marker >>
          <== the keyframe becomes selected (highlighted cyan)
          <== the playhead moves to that keyframe's time

      # click to scrub
      .if the user clicks empty space within the timeline area at the 1.5 second mark >>
          <== the playhead moves to time 1.5
          <== the viewport shows the interpolated state at time 1.5
          !== any keyframe becomes selected

      # drag to reposition
      .if a keyframe is selected at time 1.0
      .if the user drags it to the 2.5 second mark >>
          <== the keyframe snaps to time 2.5
          <== the interpolation updates to reflect the new keyframe position
          <== the playhead follows the keyframe during the drag

      # snap to grid
      .if the user drags a keyframe >>
          <== the keyframe time snaps to the nearest 0.1-second interval
          !== the keyframe rests at a time like 1.37 or 2.84

      # drag boundaries
      .if the user drags a keyframe toward time -0.5 >>
          <== the keyframe stops at time 0.0
      .if the user drags a keyframe beyond the shot duration >>
          <== the keyframe stops at the shot duration

      # delete keyframe
      .if a keyframe is selected
      .if the user presses Delete >>
          <== the keyframe is removed from its track
          <== the interpolation recalculates without that keyframe

      # minimum camera keyframe enforcement
      .if the camera track has exactly one keyframe
      .if the user attempts to delete it >>
          <== the keyframe is not deleted
          !== the camera track becomes empty

      # manual keyframe creation
      .if the playhead is at time 1.0
      .if the user presses the add-camera-keyframe shortcut >>
          <== a new camera keyframe appears at time 1.0
          <== the keyframe stores the camera's current position, rotation, and focal length

      # manual object keyframe creation
      .if an object is selected within the scene
      .if the user presses the add-object-keyframe shortcut >>
          <== a new object keyframe appears at the current time on that object's track
          <== .if no track exists for that object >> a new object track is created
    ```

    **Error cases:**
    ``` python
      # no object selected for object keyframe
      .if no object is selected within the scene
      .if the user presses the add-object-keyframe shortcut >>
          !== a keyframe is created
          !== the application produces an error

      # overlapping keyframe times
      .if a keyframe already exists at time 1.0
      .if the user drags another keyframe to time 1.0 >>
          <== the dragged keyframe merges using the existing keyframe, updating its values
          !== two keyframes occupy the same time on the same track
    ```

  - ##### 1.5.4. Interpolation and playback (Feature)

    ***Smooth interpolation between keyframes. Real-time playback evaluating every track each frame. Playback stops at shot end.***

    **Functional requirements:**
    - Position values interpolate smoothly between keyframes (not linearly snapping)
    - Rotation values interpolate using shortest-path quaternion interpolation — a camera rotating from 10 degrees to 350 degrees must travel 20 degrees (through 0), not 340 degrees the long way around
    - Focal length interpolates smoothly between keyframe values
    - Scale interpolates smoothly between keyframe values (object tracks)
    - When only one keyframe exists on a track, the value is held constant (no interpolation needed)
    - When the playhead is before the first keyframe, the first keyframe's values are held
    - When the playhead is after the last keyframe, the last keyframe's values are held
    - Playback advances time at real-time speed (1 second of playback = 1 second of shot time)
    - During playback, every track (camera and every object track) is evaluated each frame
    - Playback stops when the playhead reaches the shot duration
    - The playhead returns to time 0.0 when playback completes (or optionally stays at the end — product decision)
    - Playback can be started and stopped using the Space key or the transport bar play button
    - During playback, the user cannot select or drag keyframes
    - The current time display within the transport bar updates continuously during playback

    **Expected behavior:**
    ``` python
      # smooth position interpolation
      .if camera keyframe A is at position (0, 0, 0) at time 0.0
      .if camera keyframe B is at position (10, 0, 0) at time 2.0
      .if the playhead is at time 1.0 >>
          <== the camera is near position (5, 0, 0)
          <== the motion between A and B appears smooth, not jerky

      # shortest-path rotation
      .if camera keyframe A has rotation 10 degrees (Y-axis) at time 0.0
      .if camera keyframe B has rotation 350 degrees (Y-axis) at time 2.0
      .if the playhead is at time 1.0 >>
          <== the camera rotation is near 0 degrees (Y-axis)
          !== the camera rotates 340 degrees the long way around

      # focal length interpolation
      .if camera keyframe A has focal length 24mm at time 0.0
      .if camera keyframe B has focal length 85mm at time 2.0
      .if the playhead is at time 1.0 >>
          <== the camera focal length is between 24mm and 85mm
          <== the viewport field of view changes smoothly

      # hold values beyond keyframe range
      .if the first keyframe is at time 1.0
      .if the playhead is at time 0.5 >>
          <== the track holds the first keyframe's values

      .if the last keyframe is at time 3.0
      .if the playhead is at time 4.0 >>
          <== the track holds the last keyframe's values

      # single keyframe hold
      .if a track has exactly one keyframe >>
          <== the track's values are constant regardless of playhead position

      # real-time playback
      .if the user presses play >>
          <== the playhead advances at real-time speed
          <== the viewport updates smoothly showing the animated result
          <== every camera and object tracks are evaluated simultaneously
          <== the transport bar time display updates continuously

      # playback stops at shot end
      .if the shot duration is 5.0 seconds
      .if the user presses play >>
          <== playback stops when the playhead reaches 5.0
          !== the playhead advances beyond the shot duration

      # playback locks interaction
      .if playback is active
      .if the user clicks a keyframe >>
          !== the keyframe becomes selected
          !== the playhead jumps to the keyframe's time

      # play/pause toggle
      .if the user presses Space during playback >>
          <== playback pauses
          <== the playhead remains at the current time
      ||> .if the user presses Space again >>
          <== playback resumes from the paused time

      # multi-track evaluation
      .if the camera track has keyframes
      .if two object tracks have keyframes
      .if the user presses play >>
          <== the camera animates along its keyframed path
          <== both objects animate along their keyframed paths
          <== every animations play simultaneously
    ```

    **Edge cases:**
    ``` python
      # frame rate under heavy scenes
      .if the scene contains many objects
      .if playback cannot maintain real-time speed >>
          <== playback maintains correct timing (drops frames rather than slowing down)
          <== the shot plays for the correct wall-clock duration
          !== the shot takes longer than its duration to play

      # zero-duration gap between keyframes
      .if two keyframes exist at time 1.0 and time 1.0 (merged) >>
          <== the later keyframe's values take precedence
          <== no interpolation occurs between them
    ```

  - ##### 1.5.5. Auto-keyframing (Feature)

    ***Automatically creates or updates keyframes when the user manually moves the camera or objects. Near an existing keyframe (within 0.1s): update it. Otherwise: create a new one. This is the feature that makes Fram3d feel like directing instead of animating.***

    **Functional requirements:**
    - Auto-keyframing is always active (there is no toggle — the director should never think about whether keyframing is "on")
    - When the user manually moves, rotates, or changes the focal length of the camera, a camera keyframe is created or updated at the current playhead time
    - When the user manually moves, rotates, or scales a selected object, an object keyframe is created or updated at the current playhead time
    - "Manually" means the user is directly manipulating via mouse/keyboard input — not the result of animation playback or timeline scrubbing
    - If the current time is within 0.1 seconds of an existing keyframe on the same track, that keyframe is updated using the new values instead of creating a new keyframe
    - If the current time is not within 0.1 seconds of any existing keyframe on that track, a new keyframe is created
    - Auto-keyframing does not fire during playback
    - Auto-keyframing does not fire during timeline scrubbing (clicking the timeline to move the playhead)
    - Auto-keyframing does not fire when the system is evaluating animations (restoring state from keyframes)
    - Change detection thresholds prevent micro-movements from triggering keyframes:
      - Position: movement greater than 0.001 units
      - Rotation: rotation greater than 0.01 degrees
      - Focal length: change greater than 0.01mm
      - Scale: change greater than 0.001 units
    - When auto-keyframing creates the first keyframe on an object track, the track appears within the timeline

    **Expected behavior:**
    ``` python
      # basic camera auto-keyframe creation
      .if the playhead is at time 2.0
      .if no camera keyframe exists near time 2.0
      .if the user dollies the camera forward >>
          <== a new camera keyframe is created at time 2.0
          <== the keyframe stores the camera's new position, current rotation, and current focal length
          <== a yellow marker appears on the camera track at time 2.0

      # basic object auto-keyframe creation
      .if the playhead is at time 1.5
      .if an object is selected
      .if no object keyframe exists near time 1.5
      .if the user drags the object to a new position >>
          <== a new object keyframe is created at time 1.5
          <== the keyframe stores the object's new position, current rotation, and current scale
          <== a green marker appears on the object's track at time 1.5

      # updating an existing nearby keyframe
      .if a camera keyframe exists at time 2.0
      .if the playhead is at time 2.05 (within 0.1s)
      .if the user pans the camera >>
          <== the existing keyframe at time 2.0 is updated using the new rotation
          !== a new keyframe is created at time 2.05

      # creating a new keyframe when not near existing
      .if a camera keyframe exists at time 1.0
      .if the playhead is at time 2.5 (more than 0.1s away)
      .if the user pans the camera >>
          <== a new camera keyframe is created at time 2.5
          <== the keyframe at time 1.0 is unchanged

      # first object keyframe creates track
      .if no object track exists for "Chair"
      .if the user selects "Chair" and moves it >>
          <== an object track labeled "Chair" appears within the timeline
          <== a green keyframe marker appears at the current time

      # focal length change triggers auto-keyframe
      .if the playhead is at time 1.0
      .if the user adjusts the focal length >>
          <== a camera keyframe is created or updated at time 1.0
          <== the keyframe stores the new focal length along using current position and rotation
    ```

    **Critical: distinguishing manual input from animation evaluation:**
    ``` python
      # animation playback must not create keyframes
      .if playback is active
      .if the camera moves due to animation evaluation >>
          !== a new keyframe is created
          !== any existing keyframe is modified

      # scrubbing must not create keyframes
      .if the user clicks the timeline to scrub to time 3.0
      .if the camera and objects update to reflect time 3.0 >>
          !== a new keyframe is created
          !== any existing keyframe is modified

      # selecting a keyframe must not create keyframes
      .if the user clicks a keyframe at time 1.0
      .if the viewport updates to show the state at time 1.0 >>
          !== a new keyframe is created at time 1.0
          !== the clicked keyframe's values are overwritten

      # evaluating after keyframe drag must not create keyframes
      .if the user drags a keyframe from time 1.0 to time 2.0
      .if the system re-evaluates the animation >>
          !== additional keyframes are created as a side effect of the re-evaluation

      # switching shots must not create keyframes
      .if the user switches from Shot_01 to Shot_02
      .if the viewport updates to show Shot_02's initial state >>
          !== keyframes are created within Shot_01 or Shot_02
    ```

    **Critical: threshold and noise filtering:**
    ``` python
      # micro-movements below threshold are ignored
      .if the camera position changes by 0.0005 units (below 0.001 threshold) >>
          !== a keyframe is created or updated

      # rotation below threshold is ignored
      .if the camera rotation changes by 0.005 degrees (below 0.01 threshold) >>
          !== a keyframe is created or updated

      # accumulated small movements that cross threshold do trigger
      .if the camera has moved a total of 0.002 units from its last keyframed position (above 0.001 threshold) >>
          <== a keyframe is created or updated

      # mouse hover or accidental brush does not trigger
      .if the user's mouse moves across the viewport without holding any modifier key >>
          !== a keyframe is created
          !== the camera position changes
    ```

    **Critical: interaction with playback state transitions:**
    ``` python
      # stopping playback does not trigger auto-keyframe
      .if playback is active
      .if the user presses Space to stop playback
      .if the camera is at an interpolated position >>
          !== a keyframe is created at the stop position
          <== the camera remains at the interpolated position until the user manually moves it

      # starting playback from a modified position
      .if the user has manually moved the camera (creating a keyframe)
      .if the user presses play >>
          <== playback begins from the current time
          <== the manually-created keyframe is included within the animation

      # resuming after pause — movement creates keyframe
      .if playback was paused at time 2.0
      .if the user manually moves the camera >>
          <== a keyframe is created or updated at time 2.0
      ||> .if the user presses play >>
          <== playback resumes using the new keyframe incorporated
    ```

    **Edge cases:**
    ``` python
      # rapid sequential movements
      .if the user moves the camera, pauses briefly, then moves it again
      .if both movements occur while the playhead is within 0.1s of the same keyframe >>
          <== the same keyframe is updated both times
          !== two separate keyframes are created

      # object deselected mid-movement
      .if the user is dragging an object
      .if the user clicks empty space (deselecting the object) before releasing the drag >>
          <== the keyframe for the drag-in-progress is still committed
          !== the partial movement is lost

      # auto-keyframe at time 0.0 on new shot
      .if a new shot is created using an initial camera keyframe at time 0.0
      .if the user immediately moves the camera (playhead still at 0.0) >>
          <== the existing keyframe at time 0.0 is updated (within 0.1s threshold)
          !== a second keyframe is created at time 0.0

      # auto-keyframe near shot boundaries
      .if the shot duration is 5.0
      .if the playhead is at time 4.95
      .if a keyframe exists at time 5.0
      .if the user moves the camera >>
          <== the keyframe at time 5.0 is updated (within 0.1s threshold)
    ```

  - ##### 1.5.6. Camera path visualization (Feature)

    ***Render the camera's dolly track as a smooth spline in 3D space, showing keyframe positions as nodes along the path. Toggleable visibility.***

    **Functional requirements:**
    - The camera path is rendered as a smooth curve (spline) connecting keyframe positions in 3D space
    - Keyframe positions are shown as visible nodes (dots or spheres) along the path
    - The path shows the direction of camera travel (e.g., arrow indicators or graduated opacity)
    - The path is visible within the 3D viewport, rendered within world space (not a UI overlay)
    - The path updates in real time as keyframes are added, moved, or deleted
    - The path updates in real time as auto-keyframing creates or modifies keyframes
    - Path visibility is toggled by the user (keyboard shortcut or UI button)
    - The path is hidden by default
    - When visible, the path does not obscure critical scene elements (rendered using semi-transparency or a thin line)
    - The path represents the camera's position trajectory only — it does not visualize rotation or focal length changes
    - When only one camera keyframe exists, no path is drawn (a single node is shown)
    - The camera path corresponds to the current shot only — switching shots updates the path

    **Expected behavior:**
    ``` python
      # toggling visibility
      .if the user enables camera path visualization >>
          <== a smooth curve appears within the 3D viewport connecting camera keyframe positions
          <== nodes appear at each keyframe position along the curve
      ||> .if the user disables it >>
          <== the path and nodes are no longer visible

      # path reflects keyframes
      .if three camera keyframes exist at positions A, B, and C >>
          <== the path curves smoothly from A through B to C
          <== three nodes are visible at positions A, B, and C

      # real-time update on keyframe change
      .if the path is visible
      .if the user creates a new camera keyframe via auto-keyframing >>
          <== the path immediately updates to include the new keyframe position
          <== a new node appears at the keyframe position

      # real-time update on keyframe drag
      .if the path is visible
      .if the user drags a keyframe to a different time (changing interpolation) >>
          <== the path shape updates to reflect the new timing

      # real-time update on camera movement
      .if the path is visible
      .if the user moves the camera (updating a keyframe via auto-keyframing) >>
          <== the node at that keyframe moves to the new position
          <== the path curve updates smoothly

      # single keyframe — no path
      .if only one camera keyframe exists
      .if path visualization is enabled >>
          <== a single node is visible at the keyframe position
          !== a path or curve is drawn

      # direction indication
      .if the path is visible
      .if multiple keyframes exist >>
          <== the path indicates the direction of camera travel
          <== the user can distinguish the start from the end of the path

      # shot switching
      .if the path is visible for Shot_01
      .if the user switches to Shot_02 >>
          <== the path updates to show Shot_02's camera keyframe positions
          !== the path from Shot_01 remains visible

      # hidden by default
      .if the application starts >>
          <== the camera path is not visible
    ```

    **Edge cases:**
    ``` python
      # two keyframes at same position but different times
      .if two camera keyframes have the same position but different rotation/focal length >>
          <== the path shows a single node or overlapping nodes at that position
          !== the path displays erratic behavior

      # keyframes very close together in space
      .if two keyframes are separated by less than 0.01 units >>
          <== the path does not produce visual artifacts
          <== the curve remains smooth
    ```
