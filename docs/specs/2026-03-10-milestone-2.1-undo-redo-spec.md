# Milestone 2.1: Undo / Redo — Specification

**Date**: 2026-03-10
**Milestone**: 2.1
**Status**: Draft

---

- ### 2.1. Undo / Redo (Milestone)

  Every user action that changes the state of the project must be reversible. Undo is not a convenience feature — it is the safety net that lets directors experiment freely. Without reliable undo, users develop anxiety about every click, and the tool becomes adversarial instead of creative.

  Undo in an animation tool is fundamentally harder than undo in a text editor or drawing app. Actions have temporal context (which shot, which time on the timeline), spatial context (which object, which axis), and cascading side effects (auto-keyframing creates keyframes the user never explicitly requested). The undo system must handle all of these correctly, or it will destroy trust faster than any missing feature.

  The existing codebase has an `ICommand` interface with `Execute()` / `Undo()` / `Redo()` methods scaffolded across all timeline commands. None of the `Undo()` or `Redo()` implementations are wired up — they throw `NotImplementedException`. This milestone completes that infrastructure and connects it to a functioning undo manager.

  Selection changes are not undoable.

  *Blocked by: 1.5 (keyframe animation — the majority of undoable actions involve keyframes, tracks, and timeline state)*

  - ##### 2.1.1. Undo stack (Feature)

    ***A global, ordered history of every user action, navigable backward with Cmd+Z and forward with Cmd+Shift+Z.***

    **Functional requirements:**
    - Cmd+Z undoes the most recent action, restoring the project to its state before that action
    - Cmd+Shift+Z redoes the most recently undone action, restoring the project to its state after that action
    - The undo stack records every user-initiated state change (see "Undoable actions" below)
    - The redo stack is cleared whenever the user performs a new action after undoing
    - The undo stack has a maximum depth (minimum 100 actions) — oldest entries are discarded when the limit is exceeded
    - Undo and redo update all affected UI immediately: viewport, timeline, sequencer, keyframe editor, camera info HUD
    - When the stack is empty, Cmd+Z does nothing (no error, no feedback beyond the action being unavailable)
    - When the redo stack is empty, Cmd+Shift+Z does nothing
    - Undo stack survives save — saving does not clear the stack. After saving, Cmd+Z still undoes the most recent action.
    - Cross-shot undo does not auto-switch the active shot. If the user is in Shot 3 and undoes an action from Shot 1, the undo applies but the view stays on Shot 3.
    - Auto-keyframe + object movement is one compound undo step. Undoing a move that created a keyframe removes both the position change and the keyframe in one step.
    - Object selection and deselection are NOT undoable. They do not appear in the undo stack.
    - Restoring a deleted shot via undo does NOT switch the active shot to the restored one.

    **Undoable actions — the complete list:**

    Every action in this list is a single undo step. Actions not in this list are not recorded.

    *Camera manipulation:*
    - Moving the camera (pan, tilt, dolly, truck, crane, orbit — each continuous drag or scroll gesture is one step, not each frame of movement)
    - Rolling the camera (one scroll gesture = one step)
    - Dolly zoom (one scroll gesture = one step)
    - Changing focal length (one scroll gesture or one preset selection = one step)
    - Resetting camera to default position

    *Object manipulation:*
    - Moving an object via gizmo (one complete drag = one step)
    - Rotating an object via gizmo (one complete drag = one step)
    - Scaling an object via gizmo (one complete drag = one step)
    - Duplicating an object

    *Keyframe operations:*
    - Adding a camera keyframe (manual)
    - Adding an object keyframe (manual)
    - Deleting a keyframe
    - Dragging a keyframe to a new time (one complete drag = one step)
    - Auto-keyframe creation (each auto-created keyframe is its own step — see "Auto-keyframing and undo" below)
    - Auto-keyframe update (updating an existing keyframe's values via auto-keyframing is one step)

    *Shot operations:*
    - Adding a shot
    - Deleting a shot (captures full shot state for restoration)
    - Reordering shots (one complete drag-and-drop = one step)
    - Changing shot duration

    *Not undoable (navigation and view state):*
    - Switching between shots (selecting a different shot in the sequencer)
    - Scrubbing the timeline (moving the playhead)
    - Starting or stopping playback
    - Toggling overlays, frame guides, or camera shake
    - Changing aspect ratio
    - Selecting or deselecting objects
    - Changing gizmo mode (translate/rotate/scale)

    **Expected behavior:**
    ``` python
      # basic undo
      .if the user moves the camera
      .if the user presses Cmd+Z >>
          <== the camera returns to its position before the move
          <== the viewport updates immediately

      # basic redo
      .if the user moves the camera
      .if the user presses Cmd+Z
      .if the user presses Cmd+Shift+Z >>
          <== the camera returns to the moved position
          <== the viewport updates immediately

      # redo stack clears on new action
      .if the user moves the camera to position A
      .if the user moves the camera to position B
      .if the user presses Cmd+Z
      .if the user moves the camera to position C >>
          <== the camera is at position C
          !== pressing Cmd+Shift+Z restores position B
          <== the redo stack is empty

      # multiple sequential undos
      .if the user performs action A then B then C
      .if the user presses Cmd+Z three times >>
          <== the first Cmd+Z undoes C
          <== the second Cmd+Z undoes B
          <== the third Cmd+Z undoes A
          <== the project state matches the state before any actions

      # undo past the stack bottom
      .if the user has performed exactly two actions
      .if the user presses Cmd+Z three times >>
          <== the first two presses undo both actions
          <== the third press does nothing
          !== the application produces an error or unexpected behavior

      # stack depth limit
      .if the user has performed 100 actions
      .if the user performs one more action >>
          <== the newest action is added to the stack
          <== the oldest action is discarded
          <== the user can undo 100 times but not 101

      # undo stack survives save
      .if the user performs 5 actions
      .if the user saves the project >>
          <== the undo stack still contains all 5 actions
      ||> .if the user presses Cmd+Z >>
          <== the most recent action is undone
          !== the undo stack was cleared by the save

      # cross-shot undo does not auto-switch the active shot
      .if the user is in Shot 1 and moves the camera
      .if the user switches to Shot 3
      .if the user presses Cmd+Z >>
          <== the camera move in Shot 1 is undone
          <== the active shot remains Shot 3
          !== the view switches to Shot 1

      # auto-keyframe + object movement is one compound undo step
      .if auto-keyframing is active
      .if the user moves an object via gizmo
      .if the system auto-creates a keyframe for the object >>
          <== one undo step is recorded
          <== that step encompasses both the object position change and the new keyframe
      ||> .if the user presses Cmd+Z >>
          <== the object returns to its previous position
          <== the auto-created keyframe is removed
          !== the keyframe remains after the position is undone
          !== two separate Cmd+Z presses are required

      # selection and deselection are not undoable
      .if the user selects Object_A
      .if the user moves Object_A
      .if the user deselects Object_A
      .if the user presses Cmd+Z >>
          <== the move is undone (Object_A returns to its previous position)
          !== the deselection is undone (selection is not in the undo stack)
          !== two Cmd+Z presses are required to reach the move

      # restoring a deleted shot via undo does not switch the active shot
      .if the sequence is Shot_01, Shot_02, Shot_03
      .if the user deletes Shot_02
      .if the user is now viewing Shot_01
      .if the user presses Cmd+Z >>
          <== Shot_02 is restored to its original position between Shot_01 and Shot_03
          <== the sequencer UI updates to show all three shots
          <== the active shot remains Shot_01
          !== the active shot switches to Shot_02
    ```

  - ##### 2.1.2. Command pattern completion (Feature)

    ***Wire every user action through the command infrastructure so that each action captures enough state to reverse itself completely.***

    **Functional requirements:**
    - Every undoable action listed in 2.1.1 is executed through the command system
    - Each command captures the prior state necessary to reverse itself at the moment of execution
    - Undo restores the exact prior state — not an approximation, not a recalculation
    - Redo re-applies the exact change — not a re-execution that might produce different results
    - Commands that affect the viewport (camera moves, object transforms) restore the visual state immediately on undo/redo
    - Commands that affect the timeline (keyframe add/remove/move, shot add/delete) restore the timeline UI state immediately on undo/redo
    - Commands that affect the sequencer (shot add/delete/reorder/duration) restore the sequencer UI state immediately on undo/redo
    - Scroll gesture coalescing: rapid scroll inputs within a configurable inactivity timeout (1000ms) are coalesced into a single undo step. After 1000ms of no scroll events, the undo system treats the scroll as complete. The intent is to allow undoing from where the user "settles" — not from arbitrary mid-scroll positions.

    **Expected behavior:**
    ``` python
      # command captures prior state
      .if the camera is at position (0, 1.6, -5)
      .if the user dollies the camera forward >>
          <== the command records (0, 1.6, -5) as the prior position
          <== the command records the new position as the executed state
      ||> .if the user presses Cmd+Z >>
          <== the camera returns to exactly (0, 1.6, -5)
          !== the camera returns to a slightly different position due to floating point drift

      # undo restores deleted shot completely
      .if a shot has 4 keyframes, a 7 second duration, and the name "Shot_03"
      .if the user deletes the shot
      .if the user presses Cmd+Z >>
          <== the shot reappears in its original position in the sequence
          <== the shot has all 4 keyframes at their original times and values
          <== the shot has a 7 second duration
          <== the shot has the name "Shot_03"
          !== any keyframe data is lost
          !== the shot appears at the end of the sequence instead of its original position

      # undo restores reordered shots
      .if the sequence is Shot_01, Shot_02, Shot_03
      .if the user drags Shot_03 to the first position
      .if the user presses Cmd+Z >>
          <== the sequence is Shot_01, Shot_02, Shot_03
          !== the sequence remains Shot_03, Shot_01, Shot_02

      # undo keyframe drag restores original time
      .if a keyframe is at time 1.0s
      .if the user drags it to time 2.5s
      .if the user presses Cmd+Z >>
          <== the keyframe returns to time 1.0s
          <== the timeline UI updates to show the keyframe at 1.0s
          <== the animation evaluates correctly with the keyframe at 1.0s

      # undo object duplication removes the duplicate
      .if the scene has objects A and B
      .if the user selects A and presses Ctrl+D
      .if the user presses Cmd+Z >>
          <== the duplicated object is removed from the scene
          <== the scene has only objects A and B
          !== the original object A is affected

      # undo focal length change restores previous value
      .if the focal length is 50mm
      .if the user changes focal length to 85mm
      .if the user presses Cmd+Z >>
          <== the focal length returns to 50mm
          <== the camera info HUD shows 50mm
          <== the viewport FOV matches 50mm
    ```

## Edge Cases

These are the scenarios where undo systems typically break. Each one must be handled correctly.

### Continuous gesture coalescing

***A single drag or scroll gesture produces one undo step, not hundreds.***

When the user drags an object, each frame of the drag updates the object's position. If each frame update were recorded as a separate undo step, a simple drag-and-drop would consume dozens of undo slots and require dozens of Cmd+Z presses to reverse. The same applies to camera movements driven by mouse drag or scroll wheel.

The boundary of a gesture is defined by its input events: mouse-down to mouse-up for drags, first scroll tick to last scroll tick (with a 1000ms inactivity timeout) for scroll gestures.

``` python
  # drag coalesces into one step
  .if the user clicks an object and drags it across the screen
  .if the drag involves 60 frames of position updates
  .if the user releases the mouse >>
      <== exactly one undo step is recorded for the entire drag
      !== 60 undo steps are recorded
  ||> .if the user presses Cmd+Z >>
      <== the object returns to its position before the drag started
      !== the object moves back by one frame of the drag

  # scroll coalesces into one step
  .if the user scrolls the mouse wheel to dolly the camera
  .if the scroll involves 8 discrete scroll ticks over 0.5 seconds >>
      <== one undo step is recorded for the entire scroll gesture
  ||> .if the user pauses for long enough that the gesture is considered complete
  .if the user scrolls again >>
      <== a new, separate undo step is recorded for the second gesture

  # interrupted drag still records one step
  .if the user begins dragging an object
  .if the drag is interrupted (e.g., window loses focus, Escape pressed) >>
      <== one undo step is recorded capturing the movement up to the interruption point
      !== the drag is silently discarded without recording
      !== two partial undo steps are created
```

### Auto-keyframing and undo

***Auto-keyframing is the most dangerous interaction with undo because the system creates keyframes the user never explicitly asked for.***

When auto-keyframing creates or updates a keyframe as a side effect of moving the camera or an object, the movement and the auto-keyframe are causally linked. When the user moves the camera, two things happen: (1) the camera moves, and (2) a keyframe is created or updated. These are recorded as a single compound undo step — undoing reverses both the movement and the keyframe, because the keyframe only exists as a consequence of the movement.

``` python
  # auto-keyframe created during camera move — single compound undo
  .if auto-keyframing is active
  .if the user dollies the camera forward
  .if the system auto-creates a keyframe at the current time >>
      <== one undo step is recorded
      <== that step encompasses both the camera position change and the new keyframe
  ||> .if the user presses Cmd+Z >>
      <== the camera returns to its previous position
      <== the auto-created keyframe is removed
      <== the timeline UI no longer shows that keyframe

  # auto-keyframe updated existing keyframe — single compound undo
  .if auto-keyframing is active
  .if a camera keyframe already exists at the current time
  .if the user pans the camera >>
      <== one undo step is recorded
      <== that step encompasses both the camera rotation change and the keyframe value update
  ||> .if the user presses Cmd+Z >>
      <== the camera returns to its previous rotation
      <== the keyframe values revert to what they were before the pan
      !== the keyframe is deleted (it existed before the action)

  # multiple sequential auto-keyframes from separate gestures
  .if auto-keyframing is active
  .if the user dollies the camera (gesture completes, keyframe A created)
  .if the user pans the camera (gesture completes, keyframe B created or A updated) >>
      <== two separate undo steps are recorded
  ||> .if the user presses Cmd+Z >>
      <== only the second gesture and its keyframe effect are undone
      <== the first gesture and keyframe A remain
  ||> .if the user presses Cmd+Z again >>
      <== the first gesture and keyframe A are undone

  # auto-keyframe near existing keyframe updates it — undo restores original values
  .if auto-keyframing is active
  .if a keyframe exists at t=1.0s with position (0, 1.6, -5)
  .if the playhead is at t=1.05s (within the 0.1s near-keyframe threshold)
  .if the user moves the camera to (0, 1.6, -3) >>
      <== the existing keyframe at t=1.0s is updated to (0, 1.6, -3)
  ||> .if the user presses Cmd+Z >>
      <== the keyframe at t=1.0s reverts to (0, 1.6, -3) — wait, no:
      <== the keyframe at t=1.0s reverts to its original values (0, 1.6, -5)
      <== the camera position reverts to (0, 1.6, -5)
```

### Undo during playback

***Playback is a read-only traversal of existing state. Undo during playback must not corrupt the animation.***

``` python
  # undo while playing
  .if the user is playing back the current shot
  .if the user presses Cmd+Z >>
      <== playback stops immediately
      <== the most recent action is undone
      <== the viewport shows the result of the undo at the current playhead position
      !== playback continues with corrupted state
      !== the undo is silently ignored

  # redo while playing
  .if the user is playing back the current shot
  .if the user presses Cmd+Shift+Z >>
      <== playback stops immediately
      <== the most recent undo is redone
      <== the viewport shows the result of the redo at the current playhead position
```

### Undo across shots

***The undo stack is global across all shots. Undo does not care which shot is currently active — it undoes the most recent action regardless.***

This means the user can be viewing Shot 3, press Cmd+Z, and undo an action they performed in Shot 1. This is correct and expected — the alternative (per-shot undo stacks) would be disorienting because users would have to remember which shot they were in when they made each mistake.

``` python
  # undo action from a different shot
  .if the user is in Shot 1 and moves the camera
  .if the user switches to Shot 2 and adds a keyframe
  .if the user switches to Shot 3
  .if the user presses Cmd+Z >>
      <== the keyframe added in Shot 2 is removed
      !== the last action in Shot 3 is undone (there was no action in Shot 3)
      !== the camera move in Shot 1 is undone (it was not the most recent action)

  # cross-shot undo does not switch the active shot
  .if the user is viewing Shot 3
  .if the most recent action was in Shot 1
  .if the user presses Cmd+Z >>
      <== the action in Shot 1 is undone
      <== the active shot remains Shot 3
      !== the active shot switches to Shot 1

  # undo shot deletion while viewing a different shot
  .if the sequence is Shot_01, Shot_02, Shot_03
  .if the user deletes Shot_02
  .if the user is now viewing Shot_01
  .if the user presses Cmd+Z >>
      <== Shot_02 is restored to its original position between Shot_01 and Shot_03
      <== the sequencer UI updates to show all three shots
      <== the active shot remains Shot_01
      !== the active shot switches to Shot_02

  # undo shot reorder while viewing a different shot
  .if the sequence is Shot_01, Shot_02, Shot_03
  .if the user drags Shot_03 to the first position (now Shot_03, Shot_01, Shot_02)
  .if the user switches to Shot_01
  .if the user presses Cmd+Z >>
      <== the sequence reverts to Shot_01, Shot_02, Shot_03
      <== the user remains on Shot_01 (it was not affected by the undo)
```

### Undo and the global object timeline

***Object keyframes live on a single global timeline (1.4.3). Undoing an object edit affects every shot whose time range includes that keyframe or is affected by its interpolation.***

``` python
  # undo object move affects all shots through interpolation
  .if the user keyframes a table at (5, 0, 0) at t=5s on the global timeline
  .if this keyframe affects the interpolation visible in Shot_01 (0–5s) and Shot_02 (5–10s)
  .if the user presses Cmd+Z >>
      <== the keyframe is removed from the global timeline
      <== Shot_01 and Shot_02 both update to reflect the changed interpolation
      <== this is automatic — the global timeline is the single source of truth

  # undo link/unlink event
  .if the user links "Sword" to "Character" at t=7s on the global timeline
  .if the user presses Cmd+Z >>
      <== the link event is removed from the global timeline
      <== "Sword" is independent at t=7s and beyond
      <== any shot covering t=7s reflects the change
```

### Undo of shot duration change

***Changing shot duration can strand keyframes beyond the new endpoint. Undo must restore both the duration and any affected keyframes.***

``` python
  # shorten duration then undo
  .if a shot is 10 seconds long
  .if keyframes exist at t=3s, t=7s, and t=9s
  .if the user changes the duration to 5 seconds >>
      <== keyframes at t=7s and t=9s are either removed or clamped (per 1.5 behavior)
  ||> .if the user presses Cmd+Z >>
      <== the duration reverts to 10 seconds
      <== all three keyframes are restored at their original times with original values
      !== the keyframes at t=7s and t=9s are permanently lost
```

### Undo when undo stack contains commands for deleted objects

***If an object is deleted and later undo steps reference that object, those steps must handle the missing object gracefully.***

``` python
  # undo past a deletion
  .if the user moves Object_A to position (3, 0, 0)
  .if the user deletes Object_A
  .if the user presses Cmd+Z >>
      <== Object_A is restored (the deletion is undone)
  ||> .if the user presses Cmd+Z >>
      <== the move is undone, Object_A returns to its original position
      <== both undos work correctly in sequence

  # redo deletion after undoing it
  .if the user moves Object_A to position (3, 0, 0)
  .if the user deletes Object_A
  .if the user presses Cmd+Z (restores Object_A)
  .if the user presses Cmd+Shift+Z (re-deletes Object_A) >>
      <== Object_A is deleted again
      <== the scene state matches what it was after the original deletion
```

### Rapid undo

***The user may press Cmd+Z rapidly or hold it down. Each press must process one undo step reliably, and the system must not skip, double-process, or lag behind.***

``` python
  # rapid sequential undos
  .if the user has performed 10 actions
  .if the user presses Cmd+Z rapidly 5 times >>
      <== exactly 5 actions are undone, in reverse order
      !== actions are skipped due to input debouncing
      !== the same action is undone twice
      <== the viewport updates after each undo (not just after the sequence)

  # held Cmd+Z (key repeat)
  .if the user holds Cmd+Z and the OS generates key repeat events >>
      <== each key repeat event undoes one additional action
      <== the viewport updates smoothly as actions are unwound
```

## Error Cases

``` python
  # undo with empty stack
  .if no actions have been performed
  .if the user presses Cmd+Z >>
      <== nothing happens
      !== the application throws an error
      !== the application crashes

  # redo with empty redo stack
  .if no actions have been undone
  .if the user presses Cmd+Shift+Z >>
      <== nothing happens
      !== the application throws an error

  # undo/redo does not corrupt project state
  .if the user performs 50 actions
  .if the user undoes all 50
  .if the user redoes all 50 >>
      <== the project state matches what it was after the original 50 actions
      !== floating point drift accumulates across undo/redo cycles
      !== keyframes are lost, duplicated, or shifted in time
      !== shot order is different
      !== object positions have drifted
```

## Out of Scope

These are explicitly not part of this milestone:

- **Undo history panel / UI**: No visual list of actions. This milestone is the engine, not a history browser.
- **Named undo steps in the UI**: Commands do not need human-readable names displayed to the user (though they may have internal labels for debugging).
- **Branching undo (undo trees)**: The redo stack clears on new actions. No branching history.
- **Selective undo**: The user cannot undo an action in the middle of the stack without undoing everything after it.
- **Undo for save/load operations**: Saving or loading a project is not undoable. Loading replaces the undo stack entirely.
- **Undo for import operations**: Importing an asset is not undoable within the undo stack (too heavy — the user can delete the asset instead).
