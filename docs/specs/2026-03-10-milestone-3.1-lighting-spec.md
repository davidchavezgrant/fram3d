- ### 3.1. Lighting (Milestone)

	*Place, aim, and adjust lights to previsualize mood and contrast. Not a rendering tool — a blocking tool for DPs.*

	A DP planning a scene thinks in terms of key, fill, back, and practicals — where the light comes from, what it motivates, and how it shapes the subject. Fram3d's lighting system exists to communicate that intent. The goal is not beautiful renders. The goal is that a DP looking at the previs can say: "Key from camera left, backlight from above, motivated by the window — I understand the plan."

	Three light types map directly to real set lighting. A directional light is the sun or a large source far away — it has direction but no position. A point light is a practical — a lamp on a table, a bare bulb in a hallway. A spot light is a focused beam — a Fresnel, a Leko, a follow spot. Each type behaves the way a DP expects it to behave.

	Lights are first-class scene elements. They are selectable, movable, and aimable using the same gizmos the user already knows from objects. They have visible representations in the viewport — icons or wireframe shapes — so the user can see where lights are even when looking at the scene from an angle where the light effect is not visible. This is how lighting diagrams work on a real set: you see the fixtures, not just the light.

	Light animation supports lighting cues within a shot — a dimming lamp, a sunrise through a window, a flickering candle. Light properties are keyframeable on the timeline alongside camera and object animation.

	*Blocked by: 1.5 (keyframe animation — light animation uses the timeline system)*

	- ##### 3.1.1. Light types (Feature)

		***Directional (sun), point (practical), and spot (focused beam) — the three lights a DP thinks in. Position and aim with the same gizmo system as objects.***

		*Related:
		- 1.3.1 (scene elements — lights are selectable scene elements)
		- 1.3.2 (transform gizmos — lights use the same gizmo system)
		- 3.1.2 (light properties — each type has different configurable properties)*

		**Functional requirements:**
		- Three light types are available: directional, point, and spot
		- The user can add lights to the scene via a menu or toolbar action
			- Adding a light places it at a default position in front of the camera
			- Each light receives an auto-generated name (e.g., "Directional 1", "Spot 2")
		- Directional light:
			- Represents a distant source (the sun, a large diffused source)
			- Has rotation (direction) but no meaningful position — it illuminates the entire scene from one direction
			- Default direction: angled downward at 45 degrees (typical key light angle)
		- Point light:
			- Represents an omnidirectional practical source (a table lamp, a bare bulb)
			- Has position and range but no direction — it radiates equally in all directions
			- Default range: enough to illuminate nearby objects without flooding the scene
		- Spot light:
			- Represents a focused directional source (a Fresnel, a follow spot)
			- Has position, direction, range, and cone angle
			- Default cone angle: 30 degrees (a moderate beam)
		- Light type is changeable after creation — a point light can be converted to a spot light or directional light without deleting and recreating
			- Converting preserves shared properties (intensity, color, name)
			- Properties that do not apply to the new type are discarded (e.g., cone angle when converting spot to point)
			- Properties that are new to the target type receive default values (e.g., cone angle gets default 30 degrees when converting point to spot)
		- Lights carry across shots via continuity, just like objects. A light placed in Shot 1 exists in all subsequent shots. Light properties propagate the same way object positions do.
		- Adding the first light to a scene instantly dims the default ambient lighting. This is an instant transition, not gradual.
		- Every light type is a scene element — selectable, hoverable, and clickable using the same interaction model as objects (1.3.1)
		- Selected lights display the same selection highlight as objects
		- Selected lights can be moved and rotated with transform gizmos (1.3.2)
			- Translate gizmo moves the light's position (point, spot) or has no effect (directional)
			- Rotate gizmo changes the light's aim direction (directional, spot) or has no effect (point)
			- Scale gizmo has no effect on any light type
		- Every light has a viewport representation — a visible icon or wireframe shape that indicates its type and direction
			- Directional: an arrow or sun icon indicating the light direction
			- Point: a sphere icon or radial wireframe at the light's position
			- Spot: a cone wireframe showing beam direction and cone angle
		- Viewport representations maintain constant screen size regardless of camera distance (same behavior as gizmos)
		- Viewport representations are visible whether or not the light is selected
		- Lights can be duplicated with Ctrl+D (same as objects, per 1.3.4)
		- Lights can be deleted
		- A scene can contain zero lights
		- A scene can contain multiple lights of any type in any combination

		**Expected behavior:**
		``` python
			# adding a light
			.if user adds a spot light via the menu >>
				<== a spot light appears in the scene in front of the camera
				<== spot light is auto-named (e.g., "Spot 1")
				<== cone wireframe visible at the light's position
				<== the light illuminates objects within its cone

			# selecting and moving a light
			.if user clicks a point light's viewport icon >>
				<== point light displays selection highlight
				<== translate gizmo appears at the light's position
				||> .if user drags the green (Y) arrow handle >>
					<== point light moves upward
					<== light illumination updates to reflect new position

			# aiming a directional light
			.if user selects a directional light >>
				<== rotate gizmo appears
				||> .if user drags the rotation ring >>
					<== the light direction changes
					<== scene illumination shifts accordingly
					<== the directional arrow icon updates to show new direction

			# aiming a spot light
			.if user selects a spot light
			.if rotate gizmo is active >>
				||> .if user rotates the spot light >>
					<== cone wireframe rotates to show new aim direction
					<== illuminated area shifts to match

			# irrelevant gizmo modes
			.if user selects a directional light
			.if translate gizmo is active >>
				||> .if user drags a translate handle >>
					<== nothing happens
					!== directional light gains a position

			.if user selects a point light
			.if rotate gizmo is active >>
				||> .if user drags a rotation ring >>
					<== nothing happens
					!== point light gains a direction

			.if user selects any light
			.if scale gizmo is active >>
				||> .if user drags a scale handle >>
					<== nothing happens
					!== light changes size or intensity from scaling

			# viewport representations always visible
			.if scene contains three lights (directional, point, spot)
			.if no light is selected >>
				<== all three viewport icons/wireframes are visible
				<== each icon indicates its light type

			# viewport representation constant size
			.if camera dollies away from a point light >>
				<== point light's viewport icon remains the same screen size
				!== icon shrinks with distance

			# duplication
			.if user selects "Spot 1"
			.if user presses Ctrl+D >>
				<== "Spot 2" appears near the original with offset
				<== "Spot 2" has selection
				<== "Spot 2" has same properties as "Spot 1"

			# type conversion
			.if user has a point light named "Practical 1" with intensity 0.8 and warm color
			.if user changes its type to spot light >>
				<== light becomes a spot light
				<== name remains "Practical 1"
				<== intensity remains 0.8
				<== color remains warm
				<== cone angle is set to the default (30 degrees)
				<== cone wireframe appears at the light's position
				!== a new light is created
				!== the original light is deleted

			.if user has a spot light with cone angle 45 degrees
			.if user changes its type to directional >>
				<== light becomes a directional light
				<== cone angle is discarded (directional has no cone)
				<== range is discarded (directional has no range)
				<== intensity and color are preserved
				<== position is discarded — directional arrow icon replaces cone wireframe

			# cross-shot continuity
			.if user places a point light "Key" in Shot 1 >>
				<== "Key" exists in Shot 1
				<== "Key" exists in Shot 2 and all subsequent shots
				<== light properties in later shots match Shot 1 unless overridden
				||> .if user modifies "Key" intensity in Shot 3 >>
					<== intensity change propagates forward from Shot 3
					<== Shots 1 and 2 retain the original intensity

			# ambient dimming on first light
			.if scene has zero user-placed lights
			.if user adds a point light >>
				<== default ambient lighting dims instantly
				!== ambient dims gradually or with animation
				<== the new light's contribution is immediately dominant
		```

		**Error cases:**
		``` python
			# deleting all lights
			.if scene contains one light
			.if user deletes that light >>
				<== scene has zero lights
				<== scene renders with default ambient illumination only
				!== application crashes or shows a black viewport

			# light inside an object
			.if user places a point light inside a closed mesh >>
				<== light still illuminates — it is not occluded by the mesh
				<== viewport icon remains visible and selectable
				!== light becomes invisible or unreachable

			# many lights
			.if scene contains 50 lights >>
				<== all 50 viewport icons are visible and selectable
				<== application remains responsive
				!== viewport becomes cluttered to the point of unusability
					(viewport icons should overlap gracefully, not fight for space)

			# type conversion with keyframes
			.if a spot light has keyframes animating cone angle
			.if user converts the spot light to a point light >>
				<== cone angle keyframes are discarded
				<== intensity, color, position, and range keyframes are preserved
				!== orphaned keyframes cause errors during playback
		```

	- ##### 3.1.2. Light properties (Feature)

		***Intensity, color, range, cone angle — the properties a DP adjusts when shaping a shot. Editable in the inspector and through direct manipulation in the viewport.***

		*Blocked by:
		- 3.1.1 (light types — properties require lights to exist)*

		*Related:
		- 3.1.3 (light animation — these properties become keyframeable)
		- 3.3 (hierarchy panel — lights appear in a separate "Lights" section)*

		**Functional requirements:**
		- Every light has the following base properties:
			- Intensity — a scalar controlling how bright the light is (0 = off, 1 = default full, values above 1 allowed for overexposure effects)
			- Color — RGB color picker plus Kelvin temperature value. DPs who think in color temperature can use Kelvin directly: 2700K candlelight, 3200K tungsten, 5600K daylight, etc. Adjusting one updates the other — the RGB picker and Kelvin field are linked.
			- The default color is warm white (approximating tungsten, ~3200K)
		- Point lights additionally have:
			- Range — the maximum distance the light reaches before falling off to zero
		- Spot lights additionally have:
			- Range — same as point light
			- Cone angle — the angular width of the beam in degrees (minimum 1, maximum 179)
			- Inner cone angle — the angular width of the full-intensity core of the beam (always less than cone angle)
		- Directional lights have no range property (they illuminate infinitely)
		- Spot light cone wireframe is visible only when the spot light is selected — not always visible
		- Lights appear in a separate "Lights" section of the hierarchy panel (when hierarchy panel exists in 3.3), not mixed in with scene objects
		- **Global HUD toggle**: a single viewport toggle hides ALL HUD elements (light viewport icons, gizmos, grid, etc.) simultaneously. This is a global on/off, not per-light or per-type. Camera overlays (aspect ratio masks, safe zones) are NOT affected by this toggle — they remain independently controlled. This lets the user quickly see a clean viewport without individually toggling each HUD element.
		- Properties are editable through two methods:
			- Inspector panel: numeric fields for intensity and angles, a color picker with linked Kelvin field for color, a slider for range
			- Direct manipulation: dragging the spot cone wireframe adjusts cone angle interactively
		- Property changes take effect immediately — the scene illumination updates in real time as the user adjusts values
		- Intensity of zero turns the light off visually but does not delete it — the viewport icon remains
		- Common color temperature presets are available:
			- Candlelight (warm orange, ~1800K / ~2700K)
			- Tungsten (warm white, ~3200K)
			- Daylight (neutral white, ~5600K)
			- Overcast (cool white, ~6500K)
			- Moonlight (cool blue)
		- Presets set both the color value and the Kelvin field — the user can further adjust after applying a preset

		**Design constraints:**
		- Color is specified via both an RGB color picker and a Kelvin temperature field — DPs can work in whichever model they prefer
		- Presets are named with the motivating source (candlelight, tungsten, daylight), not with technical values alone, but the Kelvin value is displayed alongside the name
		- Cone wireframe clutter is minimized by only showing it on the selected spot light

		**Expected behavior:**
		``` python
			# adjusting intensity
			.if user selects a point light
			.if user sets intensity to 2.0 in the inspector >>
				<== light appears brighter in the scene
				<== intensity field shows 2.0

			# adjusting color via RGB picker
			.if user selects a spot light
			.if user picks a blue color via the RGB picker >>
				<== spot light casts blue illumination
				<== color swatch in inspector shows blue
				<== Kelvin field updates to the nearest equivalent color temperature

			# adjusting color via Kelvin field
			.if user selects a spot light
			.if user types 5600 into the Kelvin field >>
				<== light color changes to daylight white
				<== RGB color picker updates to match
				<== light illumination updates in real time

			# color preset
			.if user selects a directional light
			.if user applies "Daylight" preset >>
				<== light color changes to neutral white
				<== color swatch updates accordingly
				<== Kelvin field shows 5600
				||> .if user manually shifts the color warmer >>
					<== color moves away from the preset value
					<== Kelvin field updates to reflect the new temperature
					<== light illumination updates in real time

			# adjusting range
			.if user selects a point light
			.if user increases range via inspector >>
				<== light reaches farther into the scene
				<== objects previously outside the range become illuminated

			# adjusting cone angle via inspector
			.if user selects a spot light
			.if user sets cone angle to 60 degrees >>
				<== beam widens to 60 degrees
				<== cone wireframe in viewport updates to show wider angle

			# adjusting cone angle via direct manipulation
			.if user selects a spot light
			.if user drags the edge of the cone wireframe outward >>
				<== cone angle increases
				<== inspector value updates to match
				<== beam widens in real time

			# cone wireframe visibility
			.if scene contains two spot lights "Spot 1" and "Spot 2"
			.if user selects "Spot 1" >>
				<== "Spot 1" cone wireframe is visible
				<== "Spot 2" cone wireframe is not visible
				!== both cone wireframes are visible simultaneously

			# inner cone angle
			.if user selects a spot light
			.if user sets cone angle to 45 degrees
			.if user sets inner cone angle to 20 degrees >>
				<== center of beam at full intensity within 20-degree core
				<== light falls off between 20 and 45 degrees
				<== sharp center, soft edges

			# turning a light off
			.if user sets intensity to 0 >>
				<== light produces no illumination
				<== viewport icon remains visible and selectable
				!== light is deleted
				!== viewport icon disappears

			# lights in hierarchy panel
			.if scene contains two point lights and three objects >>
				<== hierarchy panel shows a "Lights" section with the two lights
				<== hierarchy panel shows objects in a separate section
				!== lights and objects are interleaved in the same list

			# global HUD toggle hides all HUD elements
			.if user toggles the global HUD visibility off >>
				<== all HUD elements are hidden: light viewport icons, gizmos, grid, etc.
				<== camera overlays (aspect ratio masks, safe zones) remain visible — they are NOT affected by the HUD toggle
				<== lights still exist and illuminate the scene — only the visual HUD elements are hidden
				||> .if user toggles the global HUD visibility back on >>
					<== all HUD elements reappear
		```

		**Error cases:**
		``` python
			# invalid cone angles
			.if user sets inner cone angle greater than outer cone angle >>
				<== inner angle clamped to match outer angle
				!== inner exceeds outer

			# cone angle at extremes
			.if user sets cone angle to 1 degree >>
				<== very narrow beam visible
				<== cone wireframe shows narrow angle

			.if user sets cone angle to 179 degrees >>
				<== very wide beam, nearly hemispherical
				<== cone wireframe shows wide angle

			# negative intensity
			.if user attempts to set intensity below 0 >>
				<== intensity clamped to 0
				!== negative intensity applied

			# Kelvin field out of range
			.if user enters a Kelvin value outside the supported range >>
				<== value is clamped to the nearest supported bound
				<== color updates to match the clamped value
				!== arbitrary or undefined color is applied
		```

	- ##### 3.1.3. Light animation (Feature)

		***Keyframe light properties on the timeline — a dimming lamp, a sunrise, a flickering candle. Lighting cues are part of the shot, not separate from it.***

		*Blocked by:
		- 1.5 (keyframe animation — light animation uses the timeline system)
		- 3.1.1 (light types — must have lights to animate)
		- 3.1.2 (light properties — must have properties to keyframe)*

		*Related:
		- 1.5.2 (tracks and keyframes — light tracks appear alongside camera and object tracks)
		- 1.5.4 (interpolation and playback — light keyframes interpolate and evaluate during playback)
		- 1.5.5 (auto-keyframing — light property changes create keyframes automatically)*

		**Functional requirements:**
		- Light properties are keyframeable on the timeline: intensity, color, position (point, spot), rotation/direction (directional, spot), range (point, spot), cone angle (spot)
		- Each animated light has its own track in the keyframe editor
			- Light tracks are visually distinct from camera tracks (yellow) and object tracks (green)
			- Light track color: a warm color that reads as "light" (e.g., orange or amber)
		- A light keyframe captures the full state of the light at that moment: intensity, color, position, rotation, range, cone angle (as applicable to the light type)
		- Adding a keyframe to a light follows the same interaction model as camera and object keyframes:
			- Manual: user presses the add-keyframe shortcut while a light is selected
			- Auto-keyframing: changing a light property while the timeline is at a specific time creates or updates a keyframe at that time (same near-keyframe logic as objects, per 1.5.5)
		- Interpolation between light keyframes is smooth (same curve-based interpolation as camera and object animation)
			- Intensity interpolates linearly
			- Color interpolates through a perceptually smooth path (no unexpected hue jumps)
			- Position and rotation interpolate the same way as object transforms
			- Range and cone angle interpolate linearly
		- During playback, light keyframes are evaluated each frame alongside camera and object keyframes
			- Light state updates smoothly as the playhead advances
		- Scrubbing the timeline updates light state in real time
		- Lights with no keyframes maintain their static properties throughout the shot — they are not affected by the timeline
		- A light can have a single keyframe (holds that state for the entire shot) or multiple keyframes (animates between them)
		- Deleting all keyframes from a light track removes the track — the light reverts to its static properties
		- Light keyframes are selectable, draggable, and deletable using the same keyframe interaction model (1.5.3)

		**Expected behavior:**
		``` python
			# adding a light keyframe
			.if user selects a point light
			.if user presses the add-keyframe shortcut >>
				<== a light keyframe appears on the point light's track at the current time
				<== keyframe captures current intensity, color, position, and range

			# animating intensity (dimming lamp)
			.if "Lamp" point light exists
			.if user adds keyframe at t=0 with intensity 1.0
			.if user scrubs to t=3.0
			.if user sets intensity to 0.2
			.if user adds keyframe at t=3.0 >>
				<== two keyframes on the Lamp track
				||> .if user plays the shot >>
					<== lamp smoothly dims from 1.0 to 0.2 over 3 seconds
					<== scene illumination changes gradually

			# animating color (sunrise)
			.if "Sun" directional light exists
			.if user adds keyframe at t=0 with warm orange color
			.if user scrubs to t=5.0
			.if user sets color to neutral white
			.if user adds keyframe at t=5.0 >>
				||> .if user plays the shot >>
					<== light color transitions smoothly from warm orange to neutral white
					!== color jumps or flickers during transition

			# auto-keyframing
			.if a point light is selected
			.if auto-keyframing is enabled
			.if user changes the light's intensity at t=2.0 >>
				<== a keyframe is created or updated at t=2.0 on the light's track
				<== keyframe captures current light state

			# scrubbing
			.if a spot light has keyframes at t=0 (cone 30 degrees) and t=4 (cone 60 degrees)
			.if user scrubs the playhead to t=2 >>
				<== cone angle displays 45 degrees (midpoint interpolation)
				<== cone wireframe updates to show 45-degree angle

			# light track visibility
			.if scene has a camera, two objects, and one animated light >>
				<== keyframe editor shows: camera track (yellow), two object tracks (green), one light track (amber/orange)
				<== light track is visually distinguishable from other track types

			# static light unaffected by timeline
			.if "Fill" point light has no keyframes
			.if user plays the shot >>
				<== Fill light maintains its properties throughout playback
				!== Fill light state changes during playback

			# removing all keyframes
			.if a light has two keyframes
			.if user deletes both keyframes >>
				<== light track removed from keyframe editor
				<== light reverts to its current static properties
				<== light is no longer animated
		```

		**Error cases:**
		``` python
			# deleting an animated light
			.if a light has keyframes on the timeline
			.if user deletes the light >>
				<== light removed from scene
				<== light track removed from keyframe editor
				<== keyframes for that light are discarded
				!== orphaned track remains in the editor
				!== application errors on next playback

			# keyframe on a property that does not apply after type conversion
			.if a spot light has keyframes animating cone angle
			.if user converts it to a point light >>
				<== cone angle keyframes are discarded
				<== intensity, color, position, and range keyframes are preserved
				!== orphaned keyframes cause errors during playback

			# playback with many animated lights
			.if scene contains 20 lights, each with keyframes >>
				<== all 20 light tracks evaluate each frame during playback
				<== playback remains at real-time speed
				<== all lights animate smoothly
		```

- ### Default scene lighting

	**When the scene has zero user-placed lights:**
	- The scene is illuminated by a default ambient light so that objects remain visible
	- The default ambient provides flat, non-directional illumination — enough to see geometry and color, but with no mood or contrast
	- The moment the user adds their first light, the default ambient reduces instantly to a low level so user-placed lights dominate the scene — this is not a gradual fade, it is an immediate change
	- If the user deletes all lights, the default ambient returns instantly to its full level

	``` python
		# empty scene
		.if scene has zero lights >>
			<== objects are visible under default ambient illumination
			<== no directional shadows or contrast
			!== scene is completely black

		# first light added
		.if scene has zero lights
		.if user adds a spot light >>
			<== default ambient dims instantly to a low level
			!== ambient dims gradually or with a fade transition
			<== spot light's contribution is clearly visible against the dimmed ambient
			<== the spot light appears to be the dominant light source

		# last light removed
		.if scene has one light
		.if user deletes that light >>
			<== default ambient returns instantly to full level
			<== objects remain visible
			!== scene goes black
	```
