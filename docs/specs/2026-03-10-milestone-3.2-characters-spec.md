# Milestone 3.2: Characters / Actors

**Date**: 2026-03-10
**Status**: Draft
**Blocked by**: 1.5 (Keyframe animation), 1.3 (Scene management)

---

- ### 3.2. Characters / Actors (Milestone)

  Poseable, animatable humanoid figures for blocking scenes. This is where Fram3d transitions from a camera tool into a previsualization tool. Without characters, directors are framing empty rooms and furniture. With characters, they are staging performances.

  The target is at least FrameForge-level character manipulation with a substantially better interaction model. FrameForge proved that directors will pose mannequins to plan blocking -- but its interface makes even simple tasks tedious. A director should be able to drop a person into a scene, apply a pose, nudge a hand, keyframe it, and move on. Every second spent wrestling with joint controls is a second not spent on the creative decision.

  Characters are scene elements (1.3). They respond to selection, gizmos, duplication, object continuity, and keyframe animation using the same systems that handle tables and chairs. But they also introduce an entirely new interaction layer: skeletal posing. The skeleton is internal complexity that the user must never feel. They are posing a person, not operating a rig.

  This is the complexity cliff of the entire product. Inverse kinematics, pose interpolation, rig mapping, and locomotion each carry more technical risk than any feature in Projects 1 or 2 combined.

  *Blocked by: 1.5 (keyframe animation -- characters animate on the timeline), 1.3 (scene management -- characters are scene elements that respond to selection, gizmos, and duplication)*

  - ##### 3.2.1. Mannequin Placement (Feature)

    ***Drop humanoid mannequins into the scene. Generic figures at real human proportions, varied by body type. Position and rotate using the same gizmo system as every other scene element.***

    *Related:
    - 1.3.1 (scene elements -- mannequins are scene elements)
    - 1.3.2 (transform gizmos -- mannequins use standard translate/rotate/scale)
    - 1.3.4 (object duplication -- Ctrl+D duplicates a mannequin with its current pose)
    - 3.2.2 (pose library -- mannequins spawn in default standing pose)*

    **Character Set**

    The built-in character set provides mannequins with male/female body type selection and continuous height and build sliders, so directors can represent the actual cast of a scene rather than staging with identical figures. Every mannequin is a featureless humanoid -- no facial features, no clothing detail. These are blocking figures, not digital actors.

    - Users pick a male or female base, then adjust height and build on continuous sliders:
      - Height slider: continuous range from short (155cm / 5'1") to tall (195cm / 6'5")
      - Build slider: continuous range from slim to heavy
    - Male and female bases differ in proportional silhouette (shoulder-to-hip ratio, torso length, limb proportions) but remain featureless mannequins
    - Every mannequin is proportionally correct for its body type at any slider position
      - Head-to-body ratio appropriate for an adult human
      - Shoulder width, hip width, limb length scaled proportionally to height and build
    - Mannequin visual style: featureless humanoid. Smooth surface, uniform matte material, no facial features, no fingers (mitten hands), no toes
      - Visually distinguishable from scene geometry (distinct silhouette, slightly different material treatment)
      - Body type visually apparent at a glance -- a short slim mannequin and a tall heavy mannequin should read as obviously different people from across a room
    - Character models must be multi-object, grouped by body region: head, torso, arms, hands, legs, feet. This grouping supports per-region selection, material assignment, and future customization. The regions are internal structure — the user still selects and moves the character as a single unit unless in pose mode.
    - Characters are color-tintable -- the user can apply a tint color to distinguish characters visually (e.g., red character vs blue character for blocking)
    - Default color: neutral gray. Tint color overrides the default when set
    - Height values represent standing height from soles to top of head
    - No limit on number of characters -- as many as the user adds. Performance concerns deferred

    - .if >> user creates a new mannequin
      ||> .if >> user selects male or female base
        - <== mannequin displays the selected base body shape
        - <== height and build sliders appear at default middle values
        - !== mannequin shape identical between male and female at the same slider values

    - .if >> user adjusts the height slider
      - <== mannequin height changes continuously in real-time as the slider moves
      - <== proportions update smoothly -- limbs, torso, and head scale proportionally
      - !== discrete jumps between height values

    - .if >> user adjusts the build slider
      - <== mannequin build changes continuously in real-time as the slider moves
      - <== shoulder width, hip width, and overall mass adjust smoothly
      - !== discrete jumps between build values

    - .if >> user applies a tint color to a mannequin
      - <== mannequin's entire surface displays the selected tint color
      - <== tint color is preserved across shots (object continuity)
      - <== tint color is visible in both viewport and rendered output
      - !== tint color affects other mannequins in the scene

    - .if >> user places multiple mannequins without setting tint colors
      - <== all mannequins display the default neutral gray
      - <== mannequins are still distinguishable by position and body type

    **Placement**

    - Mannequins are placed into the scene from a character panel or menu
    - Placement deposits the mannequin at the scene's ground level (Y=0), feet on the ground plane
    - Default placement position: center of the current camera view projected onto the ground plane, or a fixed distance in front of the camera if the camera is aimed above the horizon

    - .if >> user places a mannequin into the scene
      - <== mannequin appears standing at ground level, feet touching the ground plane
      - <== mannequin faces toward the camera (front of body toward the camera's position at time of placement)
      - <== mannequin is in the default standing pose (3.2.2)
      - <== mannequin is immediately selected
      - <== translate gizmo appears on the mannequin
      - !== mannequin floating above or embedded in the ground

    - .if >> user places a second mannequin while the first is still selected
      - <== second mannequin appears at a distinct position (not overlapping the first)
      - <== second mannequin becomes the selected element
      - <== first mannequin loses selection

    **Scene Element Behavior**

    Mannequins inherit all scene element behaviors from 1.3:

    - .if >> user clicks a mannequin
      - <== mannequin is selected (1.3.1 selection rules apply)
      - <== gizmo appears at mannequin's root position (feet/ground contact point)

    - .if >> user drags the translate gizmo on a mannequin
      - <== entire mannequin moves as a unit -- pose is preserved during translation
      - <== mannequin maintains ground contact unless explicitly lifted on Y axis
      - !== individual joints move during whole-body translation

    - .if >> user rotates a mannequin via the rotate gizmo
      - <== entire mannequin rotates around its vertical axis (Y)
      - <== pose is preserved during rotation

    - .if >> user scales a mannequin via the scale gizmo
      - <== mannequin scales uniformly -- proportions are preserved
      - !== non-uniform scale applied (no stretching a person wider without taller)

    - .if >> user presses Ctrl+D with a mannequin selected
      - <== duplicate mannequin appears with the same body type and current pose
      - <== duplicate is an independent character -- posing one does not affect the other
      - <== duplicate receives an auto-generated name ("Character_01 (1)", etc.)

    **Naming**

    - Auto-naming follows: "Character_NN" where NN is zero-padded sequential (Character_01, Character_02, etc.)
    - User can rename via the same mechanism as shot naming (double-click, inline edit)

    **Object Continuity**

    Mannequins participate in the shot sequencer's object continuity system (1.4.3):

    - .if >> user creates a new shot while a mannequin exists in the previous shot
      - <== mannequin's full state is inherited: position, rotation, scale, and current pose
      - <== pose joints carry forward -- a character sitting at the end of Shot 1 is sitting at the start of Shot 2

    - .if >> user navigates backward to a previous shot
      - <== mannequin restores to that shot's stored state, including pose

    **Error Cases**

    - .if >> user places a mannequin while the camera is inside an enclosed space with no visible ground plane
      - <== mannequin placed at a default distance in front of the camera at Y=0
      - !== mannequin placed inside geometry
      - !== placement fails silently

    - .if >> user scales a mannequin to an extreme size (very small or very large)
      - <== scale clamps to a reasonable range (minimum ~50% of original, maximum ~200%)
      - <== mannequin remains usable for posing at any valid scale

  - ##### 3.2.2. Pose Library (Feature)

    ***Preset poses that apply with one click. Standing, sitting, walking, running, pointing, conversation, lying down -- and enough variations within each to block a real scene. Poses are starting points, not final positions.***

    *Related:
    - 3.2.1 (mannequin placement -- mannequins spawn in default standing pose)
    - 3.2.3 (custom posing -- library poses serve as starting points for manual adjustment)
    - 3.2.4 (character animation -- poses are keyframeable states)*

    The pose library exists because most blocking does not require custom posing. A director needs "person sitting at a table" and "person standing by the door," not "rotate the left shoulder 47 degrees." The library should cover the poses that appear in 80% of dramatic scenes without the director ever touching a joint.

    **Pose Categories and Contents**

    Poses are organized by category. Each category contains multiple variations to avoid the "every sitting person looks identical" problem.

    | Category | Minimum Variations | Examples |
    |----------|--------------------|----------|
    | Standing | 4 | Neutral stand, weight on one leg, arms crossed, hands in pockets |
    | Sitting | 4 | Chair upright, chair relaxed/leaning back, stool/high seat, couch/low slouch |
    | Walking | 2 | Mid-stride, starting step |
    | Running | 2 | Mid-stride, sprinting |
    | Conversation | 3 | Talking (gesturing), listening (attentive), listening (arms crossed) |
    | Pointing/Directing | 2 | Arm extended pointing, directing/beckoning |
    | Lying Down | 2 | Supine (face up), on side |
    | Action | 3 | Crouching, reaching up (high shelf), leaning against wall |
    | Emotional | 3 | Head in hands (grief), celebratory (arms up), defensive (recoiling) |

    - Minimum total: 25 poses across all categories
    - Every pose must look natural at rest -- no stiff joints, no T-pose artifacts, no limbs at mathematically perfect angles
    - Poses are authored for the Average Medium body type and adapt to other body types proportionally

    **Applying Poses**

    - .if >> user selects a mannequin
      ||> .if >> user clicks a pose from the library panel
        - <== mannequin transitions to the selected pose
        - <== transition is immediate (no animated blend in the viewport -- the pose snaps on)
        - <== mannequin's position and rotation are preserved -- only the skeletal pose changes
        - <== if the pose is a sitting pose, the mannequin's vertical position adjusts so the figure appears seated at a natural height (hips at approximately seat height)
        - !== mannequin teleports to a different location
        - !== mannequin's facing direction changes

    - .if >> user applies a sitting pose to a mannequin standing next to a chair
      - <== mannequin assumes the sitting pose at its current ground position
      - <== user repositions the mannequin onto the chair manually using the translate gizmo
      - !== mannequin automatically snaps to the nearest chair

    - .if >> user applies a pose to a mannequin that was previously custom-posed (3.2.3)
      - <== library pose completely replaces the custom pose
      - <== no residual joint offsets from the previous custom pose
      - !== blend between the custom pose and the library pose

    - .if >> no mannequin is selected
      ||> .if >> user clicks a pose from the library panel
        - <== nothing happens
        - !== pose applied to the most recently selected mannequin
        - !== error dialog shown

    **Custom Poses**

    User-extensible: users can save custom poses to the library for reuse across projects. Custom poses are saved locally and persist across sessions.

    - .if >> user has a mannequin in a desired pose
      ||> .if >> user triggers "Save Pose" (via button in pose mode or library panel)
        - <== a dialog prompts the user for a pose name and optional category
        - <== the current full skeletal pose (all joint rotations + hand shapes) is saved to the user's local pose library
        - <== the saved pose appears in the library panel alongside built-in poses
        - !== the mannequin's world position or rotation is saved -- only the skeletal pose

    - .if >> user saves a custom pose with a name that already exists
      - <== the system prompts to overwrite or rename
      - !== silent overwrite of the existing pose

    - .if >> user applies a custom pose to a mannequin
      - <== the custom pose behaves identically to a built-in library pose
      - <== the pose adapts to the mannequin's body type (same adaptation rules as built-in poses)
      - <== the pose can be further modified via custom posing (3.2.3)

    - .if >> user deletes a custom pose from the library
      - <== the pose is removed from the library panel
      - <== mannequins currently holding that pose are unaffected -- they retain their current joint state
      - !== built-in poses can be deleted

    - .if >> user opens Fram3d on a different session after saving custom poses
      - <== all previously saved custom poses are available in the library
      - <== custom poses persist in local storage independent of any project

    **Custom Pose Export / Import**

    Custom poses are exportable and importable in four formats: **FBX**, **glTF**, **BVH**, and a **custom JSON format** (Fram3d-only). All four formats support both import and export of custom poses. FBX, glTF, and BVH provide interoperability with external tools. The Fram3d JSON format is a lightweight option for sharing poses between Fram3d users without requiring a full 3D interchange format.

    - .if >> user exports a custom pose
      - <== the user selects an export format: FBX, glTF, BVH, or Fram3d JSON
      - <== a file is written in the chosen format containing the full skeletal pose data
      - !== proprietary format required to share poses between users (three of four formats are industry-standard)

    - .if >> user imports a pose file
      - <== the system accepts FBX, glTF, BVH, and Fram3d JSON pose files
      - <== the pose is added to the user's local pose library
      - <== the imported pose behaves identically to a locally-created custom pose

    **Pose Library Panel**

    - Poses displayed as visual thumbnails showing a silhouette in the pose
    - Thumbnails grouped by category with category headers
    - Custom poses appear in a "Custom" category (or in user-specified categories)
    - Panel is searchable by pose name (searches both built-in and custom poses)
    - Panel is collapsible/closeable -- it does not permanently consume viewport space
    - Hovering over a pose thumbnail shows the pose name as a tooltip

    - .if >> user hovers over a pose thumbnail while a mannequin is selected
      - <== tooltip shows pose name
      - !== live preview on the mannequin (hover preview is a future enhancement, not required)

    **Pose Adaptation Across Body Types**

    - .if >> user applies "Sitting Relaxed" to a Tall Slim mannequin
      - <== pose adapts proportionally -- longer limbs, adjusted joint angles
      - <== the pose reads as the same posture on the different body type
      - !== limbs clip through each other due to proportion mismatch
      - !== the pose looks identical (ignoring body shape) to the Average Medium version -- it should look natural for the body type

    - .if >> user applies a running pose to a Short Slim mannequin
      - <== stride length and limb positions scale proportionally
      - <== the pose remains recognizable as running

    **Edge Cases**

    - .if >> user applies a pose and then immediately undoes (Cmd+Z, per 2.1)
      - <== mannequin reverts to its previous pose
      - <== undo restores the complete prior joint state

    - .if >> user applies the same pose that is already active
      - <== no visible change -- mannequin stays in the same pose
      - <== this does not create an undo entry

  - ##### 3.2.3. Custom Posing (Feature)

    ***Manipulate individual body parts to achieve specific poses. Grab a hand, the arm follows naturally via inverse kinematics. Select a joint directly for precise rotation. The 20% of poses the library cannot cover.***

    *Blocked by:
    - 3.2.1 (mannequin placement -- must have a character to pose)
    - 3.2.2 (pose library -- custom posing builds on top of library poses as starting points)*

    *Related:
    - 1.3.2 (transform gizmos -- custom posing uses a distinct joint gizmo, not the standard object gizmos)
    - 3.2.4 (character animation -- custom poses are keyframeable)*

    Custom posing is for moments the library cannot anticipate: an actor reaching for a specific doorknob, two people shaking hands at a particular angle, a character bracing against a wall. The interaction model must make simple adjustments trivial (tilt the head, raise an arm) while supporting complex poses for users willing to invest the time.

    **Entering Pose Mode**

    Posing uses a dedicated interaction mode to avoid conflicts with whole-body manipulation (translate/rotate/scale). The user explicitly enters pose mode on a selected character.

    - .if >> user selects a mannequin and enters pose mode (via button, shortcut, or double-click)
      - <== mannequin displays visible joint indicators at all manipulable joint locations
      - <== standard transform gizmo (translate/rotate/scale) is replaced by the posing interface
      - <== joint indicators appear at: head, neck, upper spine, lower spine, left/right shoulder, left/right elbow, left/right wrist, left/right hip, left/right knee, left/right ankle
      - <== total manipulable joints: 16
      - !== gizmos for whole-body translate/rotate/scale shown simultaneously

    - .if >> user clicks empty space or presses Escape while in pose mode
      - <== pose mode exits
      - <== joint indicators disappear
      - <== standard gizmo reappears on the mannequin (if still selected)
      - <== the current pose is preserved

    - .if >> user selects a different scene element while in pose mode
      - <== pose mode exits on the current mannequin
      - <== new element is selected normally

    **Joint Hierarchy**

    The mannequin skeleton follows a simplified humanoid hierarchy:

    ```
    Root (hips)
      Lower Spine
        Upper Spine
          Neck
            Head
          Left Shoulder
            Left Elbow
              Left Wrist
          Right Shoulder
            Right Elbow
              Right Wrist
      Left Hip
        Left Knee
          Left Ankle
      Right Hip
        Right Knee
          Right Ankle
    ```

    - Child joints inherit parent rotation -- rotating the upper spine tilts the head and both arms
    - The root (hips) controls whole-skeleton offset within pose mode -- moving the hips raises/lowers the entire body (crouch, jump)

    **Inverse Kinematics (IK)**

    IK is the primary interaction method for limbs. When the user grabs an end effector (hand or foot), the intermediate joints (elbow/shoulder or knee/hip) solve automatically to maintain a natural chain.

    - .if >> user grabs a wrist joint indicator and drags
      - <== the wrist follows the cursor
      - <== the elbow and shoulder rotate automatically to keep the arm chain connected
      - <== the arm maintains a natural bend -- the elbow does not invert or snap to unnatural angles
      - <== the rest of the body is unaffected (torso, other arm, legs remain in place)

    - .if >> user grabs an ankle joint indicator and drags
      - <== the ankle follows the cursor
      - <== the knee and hip rotate automatically to keep the leg chain connected
      - <== the knee maintains a natural forward bend
      - !== the knee bends backward

    **IK Reach Limits**

    - .if >> user drags a wrist beyond the arm's fully extended reach
      - <== the arm extends fully straight (shoulder + upper arm + forearm at maximum length)
      - <== the wrist stops at maximum reach distance -- it does not detach or stretch
      - <== the shoulder may rotate to accommodate the direction of the pull, up to its natural range of motion
      - !== the arm stretches beyond anatomical length
      - !== the character's torso is pulled toward the hand

    - .if >> user drags an ankle beyond the leg's fully extended reach
      - <== same behavior as arm: full extension, then stop
      - <== if the foot is pulled far enough, the hip may rotate but the spine does not collapse
      - !== the character falls over or ragdolls

    **IK Pole Targets (Elbow/Knee Direction)**

    Elbow and knee direction during IK solving is a common source of unnatural results. The system must maintain plausible joint orientation.

    - .if >> user moves a wrist such that the elbow direction becomes ambiguous (e.g., arm fully extended then bent again)
      - <== elbow defaults to pointing backward/outward (natural human resting elbow direction)
      - <== elbow direction remains consistent during continuous drag -- no sudden flips
      - !== elbow snaps between pointing left and pointing right during smooth wrist movement

    - .if >> user needs to override elbow/knee direction
      - <== user can directly select and rotate the elbow or knee joint to set the preferred bend direction
      - <== subsequent IK solving on that limb respects the user's elbow/knee preference as a hint

    **Direct Joint Rotation (FK)**

    For joints where IK is not applicable or where the user wants precise control, direct rotation (forward kinematics) is available.

    - .if >> user clicks a joint indicator (not an end effector)
      - <== a rotation gizmo appears at that joint
      - <== dragging the gizmo rotates the joint and all its children
      - <== parent joints are unaffected

    - .if >> user rotates the head joint
      - <== only the head rotates (no children)
      - <== the neck and spine remain fixed

    - .if >> user rotates the upper spine
      - <== the upper spine, neck, head, and both arms rotate together
      - <== the lower spine and legs remain fixed

    - .if >> user rotates a shoulder joint
      - <== the shoulder, elbow, wrist, and hand rotate together
      - <== the spine and other arm remain fixed

    **Joint Limits**

    Every joint has rotation limits that prevent anatomically impossible poses.

    - .if >> user rotates a joint to the limit of its range of motion
      - <== rotation stops at the limit -- the joint does not rotate further
      - <== visual feedback indicates the limit has been reached (resistance, color change on the joint indicator, or damped movement near the limit)
      - !== the joint rotates through the body
      - !== the joint allows 360-degree unrestricted rotation

    - Joint limit ranges (approximate -- these define the constraint envelope, not exact degree values):
      - Head: generous range on all axes (look left/right, up/down, tilt)
      - Neck: moderate range, less than head
      - Spine segments: moderate flexion/extension, limited lateral bend and twist
      - Shoulder: wide range -- full abduction, flexion, rotation (the most mobile human joint)
      - Elbow: single-axis (flexion only, 0 to ~145 degrees), no hyperextension
      - Wrist: moderate flexion/extension and lateral deviation, moderate rotation (pronation/supination)
      - Hip: wide range but less than shoulder -- flexion, abduction, rotation
      - Knee: single-axis (flexion only, 0 to ~140 degrees), no hyperextension
      - Ankle: moderate dorsiflexion/plantarflexion, limited lateral tilt

    **Spine Posing**

    The spine is neither a pure IK chain nor a single joint. It requires a blended approach so the torso can assume natural curves (leaning forward, twisting, arching back) without requiring the user to individually rotate each vertebra.

    - .if >> user grabs the upper spine and drags
      - <== both spine segments distribute the rotation to produce a natural curve
      - <== the head and arms follow the spine movement
      - <== the hips remain planted (unless the user explicitly moves the root)
      - !== only the upper spine segment rotates while the lower spine stays rigid (this looks robotic)

    - .if >> user wants an asymmetric spine bend (twist to look over shoulder)
      - <== user can individually rotate each spine segment via FK for full control

    **Finger Posing**

    Fingers are deliberately excluded from individual joint control. The mannequin uses mitten hands (3.2.1). However, basic hand shapes are needed for silhouette reads.

    - The system provides a small set of hand shape presets:
      - Open (fingers extended)
      - Relaxed (slight natural curl)
      - Fist (closed hand)
      - Pointing (index extended, others curled)
      - Grip (as if holding a cylindrical object)
    - Hand shape is applied per-hand via a selector in pose mode
    - Hand shapes are included in the pose state -- they keyframe and interpolate with the rest of the pose

    - .if >> user changes the left hand to "Pointing" shape
      - <== left hand displays pointing shape
      - <== right hand is unchanged
      - <== hand shape is preserved when exiting pose mode

    **Prop Holding**

    Characters can hold objects (FrameForge-style). The character grips the object but finger alignment is not precise -- the object attaches to the hand area without detailed finger posing.

    - .if >> user attaches a scene object to a character's hand
      - <== the object parents to the wrist joint and moves with the hand during posing and animation
      - <== the object's position/rotation can be offset relative to the hand for alignment
      - <== the hand shape automatically switches to "Grip" (unless the user overrides)
      - !== the object snaps precisely into individual fingers
      - !== the object affects the IK solving of the arm

    - .if >> user detaches a prop from a character's hand
      - <== the object becomes an independent scene element again
      - <== the object remains at its current world position (does not snap back to its original location)
      - <== the hand shape reverts to its previous state (or "Relaxed" if no prior state)

    - .if >> user poses a character's arm while the character is holding a prop
      - <== the prop follows the hand movement
      - <== the prop's relative offset to the hand is preserved

    - .if >> character animation plays back and the character is holding a prop
      - <== the held prop automatically follows the hand bone during animation — no additional keyframing required on the prop
      - <== the prop maintains its relative offset to the hand throughout the animation

    - .if >> user keyframes a character holding a prop
      - <== the prop's attachment state is part of the character keyframe
      - <== the prop moves with the hand during pose interpolation

    **Self-Collision and Body Constraints**

    Self-collision detection is skipped for v1. Limbs can clip through the body or each other. No "partial character" mode (bust/upper body only) -- full body only.

    **Pose Reset**

    - .if >> user triggers "Reset Pose" while in pose mode
      - <== all joints return to the default standing pose (3.2.2 Standing Neutral)
      - <== this is an undoable action

    - .if >> user triggers "Reset Joint" on a specific joint
      - <== only that joint and its children return to their default rotation
      - <== the rest of the skeleton is unaffected

    **Interaction Model Summary**

    | Body Part | Primary Method | Fallback Method |
    |-----------|---------------|-----------------|
    | Wrist/Hand | IK (drag) | FK (rotate shoulder, elbow, wrist individually) |
    | Ankle/Foot | IK (drag) | FK (rotate hip, knee, ankle individually) |
    | Elbow | IK hint (set bend direction) | FK (direct rotate) |
    | Knee | IK hint (set bend direction) | FK (direct rotate) |
    | Head | FK (direct rotate) | -- |
    | Neck | FK (direct rotate) | -- |
    | Spine (upper/lower) | Drag (distributed bend) | FK (per-segment rotate) |
    | Hips/Root | Translate (move whole body up/down/forward) | FK (rotate pelvis tilt) |
    | Hands | Preset shapes | -- |

    **Edge Cases**

    - .if >> user is in pose mode on Character_01
      ||> .if >> user clicks on Character_02
        - <== pose mode exits on Character_01
        - <== Character_02 is selected normally (not in pose mode)
        - <== Character_01 retains its pose

    - .if >> two mannequins overlap and the user clicks in the overlapping region while in pose mode
      - <== the click resolves to the nearest joint indicator, not the other mannequin
      - <== if no joint indicator is near the click, pose mode exits and the frontmost mannequin is selected

    - .if >> user poses a limb such that it clips through the mannequin's own body (e.g., arm through torso)
      - <== the pose is allowed -- the system does not enforce self-collision avoidance
      - !== the system prevents the pose or snaps the limb to the body surface
      - Rationale: collision prevention would fight the user. Clipping is preferable to being unable to achieve a desired pose. The director can see the clip and adjust.

    - .if >> user poses a mannequin such that a hand clips through a scene object (table, wall)
      - <== the pose is allowed -- no collision enforcement between characters and objects
      - <== the user adjusts manually if the clip is unacceptable

    - .if >> user is in pose mode and triggers undo (Cmd+Z)
      - <== the most recent joint manipulation is undone
      - <== joint reverts to its previous rotation
      - <== undo granularity: each completed drag or rotation operation is one undo step

  - ##### 3.2.4. Character Animation (Feature)

    ***Keyframe character poses on the timeline. Interpolate between poses for blocking animation. Walk cycles and basic locomotion. Characters come alive enough to plan camera movement around their performance.***

    *Blocked by:
    - 1.5 (keyframe animation -- character animation lives on the timeline)
    - 3.2.1 (mannequin placement -- must have characters to animate)
    - 3.2.3 (custom posing -- the poses being keyframed come from the posing system)*

    *Related:
    - 1.4.3 (object continuity -- character animation state carries across shots)
    - 1.5.2 (tracks and keyframes -- characters get their own tracks)
    - 1.5.5 (auto-keyframing -- posing a character at a time creates/updates keyframes)*

    Character animation extends the existing keyframe system (1.5) to handle pose data. Where object keyframes store position/rotation/scale (6 channels), character keyframes store position/rotation/scale plus the full skeletal pose (all joint rotations plus hand shapes). This is the hardest interpolation problem in the product.

    **Character Tracks**

    - .if >> user creates a keyframe on a mannequin (manually or via auto-keyframing)
      - <== a character track appears in the keyframe editor
      - <== character tracks are visually distinct from object tracks and camera tracks (unique color -- suggested: orange or purple, distinct from camera yellow and object green)
      - <== the track is labeled with the character's name

    - .if >> a mannequin has no keyframes in the current shot
      - <== no character track appears for that mannequin
      - <== the mannequin holds its initial state for the shot (per 1.4.3)

    **Character Keyframes**

    A character keyframe captures the complete character state at a point in time:
    - World position (Vector3)
    - World rotation (Quaternion)
    - Scale (uniform float)
    - Full skeletal pose: rotation of every joint (16 joints, each a Quaternion)
    - Hand shapes (left and right, each an enum value)

    - .if >> user poses a mannequin while the shot is at time t
      - <== a character keyframe is created (or updated) at time t containing the full current state
      - <== auto-keyframing rules from 1.5.5 apply: near existing keyframe (0.1s) updates it, otherwise creates new

    - .if >> user applies a library pose (3.2.2) at time t
      - <== a character keyframe is created at time t with the library pose
      - <== this follows the same auto-keyframe rules

    - .if >> user adds a keyframe explicitly (via keyboard shortcut, per 1.6.2)
      - <== character keyframe captured at the current time with the current full state

    **Pose Interpolation**

    Interpolating between two full-body poses is the central technical challenge. A character going from standing to sitting does not linearly blend -- the knees must bend before the hips lower, the hands might reach for armrests. Perfect interpolation is not the goal; plausible intermediate poses are.

    - .if >> timeline has two character keyframes (e.g., standing at t=0, sitting at t=2)
      ||> .if >> playhead moves to t=1 (midpoint)
        - <== the mannequin displays an interpolated pose between standing and sitting
        - <== every joint rotation is interpolated individually (spherical interpolation, not linear Euler)
        - <== world position is interpolated smoothly
        - <== the interpolated pose looks plausible -- no joints inverted, no limbs passing through the body in ways worse than the keyframes themselves
        - !== the mannequin snaps between keyframes with no in-between
        - !== joints interpolate via the long arc (e.g., a 10-degree rotation should not travel 350 degrees the other way)

    - .if >> three or more keyframes exist for a character within a shot
      - <== interpolation produces a smooth path through all keyframes
      - <== the character does not overshoot or oscillate between keyframes

    - .if >> two adjacent keyframes have very similar poses (small adjustment)
      - <== interpolation is subtle and smooth
      - !== exaggerated motion between nearly identical poses

    - .if >> two adjacent keyframes have radically different poses (standing to lying down)
      - <== interpolation produces a continuous motion that passes through all intermediate joint angles
      - <== the result may not look physically realistic (no simulated getting-down motion) but should not produce grossly broken poses (limbs detaching, body folding inside-out)
      - <== minor self-clipping is acceptable (arms may briefly pass through torso during transition between radically different poses)

    **Hand Shapes**

    Hand shapes are discrete presets sufficient for blocking: open hand, fist, pointing, grip. No per-finger posing.

    **Hand Shape Interpolation**

    - .if >> keyframe A has left hand "Open" and keyframe B has left hand "Fist"
      - <== the hand shape transitions at the midpoint between the two keyframes (discrete switch, not blended)
      - <== the switch point is the temporal midpoint between the two keyframes
      - !== the hand shape blends between open and fist (these are discrete presets, not continuous)

    **Interpolation and Joint Limits**

    - .if >> interpolation between two valid poses produces an intermediate pose that would violate a joint limit
      - <== the joint is clamped to its limit for the offending frames
      - <== the interpolation resumes normally once the natural path re-enters the valid range
      - !== the joint snaps or produces discontinuous motion at the limit boundary

    **Walk Cycles and Locomotion**

    Walk cycles solve the specific problem of "this character needs to walk from point A to point B." Manually keyframing a walk is painstaking -- the user should be able to define a path and have the legs move automatically.

    - The system provides pre-baked walk cycle animations:
      - Walk (casual pace)
      - Walk (brisk/purposeful)
      - Run
      - Jog
    - Walk cycles support custom gaits: limping, sneaking, running, etc. (gait presets available from standard animation libraries)
    - Foot sliding: minimal visible sliding is acceptable for v1. Perfect foot-contact IK is not required

    - .if >> user assigns a locomotion cycle to a character
      ||> .if >> the character has position keyframes defining a path
        - <== the character moves along the keyframed path
        - <== the legs and arms animate with the assigned locomotion cycle
        - <== locomotion cycle speed adapts to the distance traveled per unit time (steps get faster or slower to match the path speed)
        - <== the character faces the direction of travel (automatically rotates along the path tangent)
        - !== the character slides without leg movement

    - .if >> user assigns a custom gait preset (e.g., limping, sneaking)
      ||> .if >> the character has position keyframes defining a path
        - <== the character moves along the keyframed path using the selected gait animation
        - <== gait speed adapts to the path speed (same rules as standard locomotion cycles)
        - <== the gait is visually distinct from the standard walk/run cycles

    - .if >> user assigns a locomotion cycle and the path speed exceeds the run animation's maximum plausible speed
      - <== the system uses the run cycle and allows some foot-sliding rather than refusing the speed
      - <== this is a graceful degradation, not an error

    - .if >> user assigns a locomotion cycle but the character has no position keyframes (stationary)
      - <== the character plays the cycle in place (walking in place)
      - <== this is a valid state -- the user may want to preview the cycle before setting a path

    - .if >> the character's path includes a stop (two position keyframes at the same location with time between them)
      - <== the character transitions from the locomotion cycle to a standing idle at the stop point
      - <== the character resumes the locomotion cycle when the path continues

    **Locomotion Transition**

    - .if >> a character is walking and reaches its final position keyframe
      - <== the locomotion cycle blends out into the character's keyframed pose at that time
      - <== if no pose keyframe exists at the end time, the character settles into a neutral standing pose

    - .if >> a character has both manual pose keyframes and a locomotion cycle active
      - <== locomotion controls the legs and root position along the path
      - <== manual pose keyframes for the upper body (spine, arms, head) override the locomotion's upper body
      - <== this enables "character walks while carrying a box" -- walk cycle on legs, custom arm pose on top

    **Playback Integration**

    - .if >> user presses play (Space) and the shot contains character keyframes
      - <== characters animate in real-time alongside camera and object animations
      - <== all tracks evaluate synchronously at each frame

    - .if >> user scrubs the timeline with character keyframes present
      - <== the mannequin updates to the interpolated pose at the scrub position
      - <== scrubbing is responsive -- pose evaluation does not cause visible lag

    **Auto-Keyframing for Characters**

    - .if >> user enters pose mode and adjusts a joint while the shot is not playing
      - <== auto-keyframing rules (1.5.5) apply to the full character state
      - <== if near an existing character keyframe (within 0.1s): the existing keyframe is updated with the new joint rotation
      - <== if not near an existing keyframe: a new character keyframe is created at the current time
      - <== the new/updated keyframe captures the complete pose, not just the modified joint

    - .if >> user translates a mannequin (whole-body move via gizmo) while character keyframes exist
      - <== auto-keyframing creates/updates a keyframe with the new position
      - <== the skeletal pose at the new keyframe matches the current pose

    **Edge Cases**

    - .if >> user deletes a character keyframe that is between two other keyframes
      - <== the interpolation recomputes between the remaining keyframes
      - <== the character may now pass through a different intermediate pose at the deleted keyframe's time

    - .if >> user deletes all keyframes for a character within a shot
      - <== character track is removed from the keyframe editor
      - <== character reverts to its initial state for the shot

    - .if >> user is in pose mode and playback starts
      - <== pose mode exits automatically
      - <== playback proceeds normally
      - !== the user can pose joints while animation is playing

    - .if >> user duplicates (Ctrl+D) a mannequin that has character keyframes in the current shot
      - <== the duplicate mannequin has no keyframes (per 1.3.4 -- animation not copied)
      - <== the duplicate holds its current pose as a static initial state
      - !== the duplicate inherits the original's animation

    - .if >> a shot has 10+ characters each with multiple keyframes
      - <== playback and scrubbing remain responsive
      - <== all characters evaluate simultaneously

  - ##### 3.2.5. Custom Character Import (Feature)

    ***Import rigged humanoid models in FBX format. Map the imported rig to Fram3d's pose and animation system. Let directors bring their own characters -- stylized, realistic, or production-specific.***

    *Blocked by:
    - 2.3.1 (model import -- FBX import pipeline must exist)
    - 3.2.1 (mannequin placement -- the built-in character system must exist before custom characters extend it)
    - 3.2.3 (custom posing -- imported characters must work with the same posing interface)*

    *Related:
    - 3.2.2 (pose library -- imported characters should support the same pose library)
    - 3.2.4 (character animation -- imported characters animate identically to built-in mannequins)*

    Custom character import is the bridge between Fram3d and production pipelines. A director working with a concept artist or a studio with established character models should be able to bring those characters into the previs tool. The fundamental challenge is rig mapping: every rigging artist names bones differently, structures hierarchies differently, and makes different assumptions about bind poses and axis conventions.

    **Import Flow**

    - .if >> user imports an FBX file containing a rigged humanoid model
      - <== the system analyzes the rig to determine if it is a valid humanoid skeleton
      - <== if the rig is recognized as humanoid: the character appears in the scene and is immediately usable with the pose system
      - <== if the rig cannot be mapped: the model imports as a static object (per 2.3.1), not as a poseable character, and the user is notified

    - .if >> the imported rig uses non-standard bone names (e.g., "L_arm_upper" instead of "LeftUpperArm")
      - <== the system attempts automatic mapping based on hierarchy structure, bone positions, and common naming conventions
      - <== if automatic mapping succeeds: the character is fully functional
      - <== if automatic mapping is ambiguous or fails: the user is presented with a mapping interface

    **Rig Mapping Interface**

    When automatic mapping cannot confidently resolve the rig, the user assists:

    - The mapping interface presents Fram3d's 16 required joints as a list
    - For each required joint, the system shows its best guess from the imported rig (if any)
    - The user can confirm, override, or clear each mapping
    - A visual preview shows the imported model with the current mapping applied -- the user can see immediately if the mapping is correct
    - The user can apply the mapping and see the character assume a test pose (e.g., the default standing pose)

    - .if >> user completes the rig mapping and confirms
      - <== the character is added to the scene as a fully poseable character
      - <== all pose library poses work on the imported character
      - <== all custom posing (IK, FK) works on the imported character
      - <== all character animation features work on the imported character

    - .if >> user cancels the rig mapping
      - <== the model imports as a static (non-poseable) object
      - <== no character track is created for it

    **Mapping Requirements**

    The imported rig must map to Fram3d's 16-joint skeleton (3.2.3). Additional bones in the imported rig are ignored for posing purposes but preserved for rendering.

    - Required mappings (all 16 must be resolved for a character to be poseable):
      - Hips (root), Lower Spine, Upper Spine, Neck, Head
      - Left Shoulder, Left Elbow, Left Wrist
      - Right Shoulder, Right Elbow, Right Wrist
      - Left Hip, Left Knee, Left Ankle
      - Right Hip, Right Knee, Right Ankle

    - .if >> the imported rig has additional bones (fingers, toes, twist bones, helper bones)
      - <== additional bones are driven by their parent joint in the mapping -- they follow along but are not individually poseable
      - <== the visual result includes the additional bone deformations, preserving the character's mesh quality

    - .if >> the imported rig has fewer bones than required (e.g., no separate neck bone)
      - <== the mapping interface flags the missing joint
      - <== the user can assign a substitute (e.g., map Neck to the same bone as Head) or leave it unmapped
      - <== unmapped joints are locked in their bind pose -- they cannot be posed
      - <== the character is still usable with reduced posing capability

    **Bind Pose and Axis Conventions**

    Imported rigs may use different bind poses (T-pose vs. A-pose) and axis conventions (Y-up vs. Z-up, different bone-local axis orientations).

    - .if >> the imported character is in a T-pose bind pose and Fram3d's internal skeleton uses a different convention
      - <== the system computes the offset between the bind pose and Fram3d's reference pose
      - <== library poses are adapted through this offset so they produce the correct visual result on the imported character
      - !== library poses produce distorted results because of bind pose differences

    - .if >> the imported character uses Z-up axis convention
      - <== the system applies the appropriate axis rotation on import
      - <== the character stands upright on the ground plane

    **Saved Mappings**

    - .if >> user imports a second character from the same source (same rig structure)
      - <== the system recognizes the rig structure and applies the previously saved mapping automatically
      - <== the user can still override if needed

    - Rig mappings are saved per-project, associated with the rig structure (not the file path)
    - .if >> user re-imports the same FBX after modifying it externally
      - <== the saved mapping is applied if the rig structure is unchanged
      - <== if the rig structure changed, the mapping process restarts

    **Visual Fidelity**

    - .if >> the imported character has textures and materials in the FBX
      - <== textures and materials are preserved on the imported character
      - <== the character renders with its original appearance, not as a gray mannequin

    - .if >> the imported character has blend shapes / morph targets
      - <== blend shapes are preserved for rendering but are not exposed for manipulation
      - <== this is a known limitation -- blend shape animation is not supported in this milestone

    **Edge Cases**

    - .if >> user imports a non-humanoid rig (quadruped, mechanical, abstract)
      - <== automatic humanoid mapping fails
      - <== the model imports as a static object
      - <== user is notified: "This model could not be recognized as a humanoid character. It has been imported as a static object."
      - !== the system attempts to force-fit a non-humanoid skeleton into the humanoid template

    - .if >> user imports an FBX with multiple rigged characters in one file
      - <== each rigged character is imported as a separate poseable character
      - <== each gets its own rig mapping (if needed)

    - .if >> user imports a character with extremely non-standard proportions (e.g., a caricature with an oversized head)
      - <== the rig mapping still works based on skeleton structure, not proportions
      - <== pose library poses may look unusual on extreme proportions but should not break
      - <== IK reach limits adapt to the actual bone lengths of the imported character

    - .if >> user imports a character and then the source FBX file is moved or deleted
      - <== the character remains in the scene -- the import is a copy, not a reference
      - !== the character disappears or shows a missing-file error

## Acceptance Criteria

### Mannequin Placement

1. User places three mannequins with different body configurations (varying base, height slider, build slider) into a scene.
   - <== three visually distinct mannequins visible at ground level
   - <== each is selectable and movable via translate gizmo
   - <== heights and builds are visually and measurably different
   - <== male and female bases produce distinct silhouettes

2. User duplicates a posed mannequin (Ctrl+D).
   - <== duplicate has the same body type, slider values, tint color, and pose
   - <== duplicate is offset from the original
   - <== posing one does not affect the other

3. User applies a tint color to a mannequin.
   - <== mannequin displays the selected tint color
   - <== other mannequins remain their original color
   - <== tint color persists across shots

### Pose Library

4. User selects a mannequin and applies "Sitting Relaxed" from the library.
   - <== mannequin assumes sitting pose immediately
   - <== mannequin's ground position is preserved
   - <== pose looks natural (no stiff limbs, no floating)

5. User applies "Walking Mid-stride" to one mannequin and "Conversation Listening" to another.
   - <== each mannequin holds its respective pose
   - <== applying a pose to one does not affect the other

6. User applies a pose, then presses Cmd+Z.
   - <== mannequin reverts to its previous pose

7. User custom-poses a mannequin, saves the pose to the library, closes and reopens the application.
   - <== custom pose appears in the library panel under "Custom" category
   - <== custom pose persists after application restart
   - <== applying the custom pose to a different mannequin works identically to built-in poses

### Custom Posing

8. User enters pose mode, grabs a wrist, and drags it to a new position.
   - <== arm follows naturally via IK -- elbow bends, shoulder rotates
   - <== rest of body is unaffected

9. User drags a wrist beyond arm's reach.
   - <== arm extends fully straight and stops -- no stretching, no body pull

10. User rotates the upper spine forward.
    - <== torso, head, and arms tilt forward together
    - <== legs remain planted

11. User poses an arm through the mannequin's torso.
    - <== pose is allowed (no collision enforcement -- self-collision skipped for v1)
    - <== the clipping is visible and the user can adjust

12. User exits pose mode by pressing Escape.
    - <== joint indicators disappear
    - <== pose is preserved
    - <== standard gizmo reappears

13. User attaches a scene object (e.g., a cup) to a character's hand.
    - <== the object follows the hand during posing and animation
    - <== the object can be offset relative to the hand for alignment
    - <== detaching the object returns it to an independent scene element

### Character Animation

14. User creates two pose keyframes: standing at t=0, sitting at t=3.
    - <== scrubbing to t=1.5 shows a mid-transition pose
    - <== the mid-transition pose is plausible (no inverted joints, no body folding)
    - <== minor self-clipping during transition between radically different poses is acceptable

15. User assigns a walk cycle and sets two position keyframes defining a 5-meter path.
    - <== the character walks from start to end with animated legs
    - <== minimal visible foot sliding is acceptable for v1
    - <== character faces the direction of travel

16. User assigns a custom gait (e.g., limping) and sets position keyframes defining a path.
    - <== the character moves along the path with the selected gait animation
    - <== the gait is visually distinct from standard walk/run

17. User starts playback with two animated characters.
    - <== both characters animate simultaneously and in sync
    - <== playback is smooth and responsive

18. User deletes all keyframes for a character.
    - <== character track disappears from keyframe editor
    - <== character holds its initial pose for the shot

### Custom Character Import

19. User imports an FBX with a standard humanoid rig.
    - <== rig is automatically mapped
    - <== character appears in scene and is poseable
    - <== pose library works on the imported character

20. User imports an FBX with non-standard bone naming.
    - <== mapping interface appears
    - <== user maps joints manually
    - <== character becomes poseable after mapping is confirmed

21. User imports a non-humanoid model (e.g., a car).
    - <== model imports as a static object
    - <== user receives clear notification
    - !== application error or crash

### Cross-Feature

22. User places a mannequin in Shot_01, poses it sitting, creates Shot_02.
    - <== mannequin in Shot_02 inherits the sitting pose
    - <== navigating back to Shot_01 shows the mannequin in Shot_01's original state

23. User has a scene with 5 characters, each with 3 pose keyframes across a 10-second shot.
    - <== playback runs at real-time with all characters animating
    - <== scrubbing remains responsive

24. User imports a custom character, applies a library pose, adjusts via custom posing, keyframes the result.
    - <== the full pipeline works: import -> library pose -> custom adjustment -> keyframe
    - <== the imported character behaves identically to a built-in mannequin throughout this workflow
