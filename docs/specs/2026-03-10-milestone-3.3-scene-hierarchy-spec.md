- ### 3.3. Scene hierarchy (Milestone)
	*Parent/child relationships between objects. Group objects, attach props to characters, build compound setups that move as a unit.*

	A director thinks in terms of "this goes with that." A table has plates on it. A character holds a prop. Passengers ride in a car. When the table moves, the plates come along — nobody expects to reposition each plate individually. Hierarchy formalizes this intuition: drag one object onto another and they move together.

	This milestone adds parenting and a hierarchy panel. It does not add grouping (selecting multiple unrelated objects and treating them as one) — that is a separate concept that may come later. Hierarchy is strictly parent/child: one parent, transforms inherited downward.

	*Blocked by: 1.3 (scene management — hierarchy extends the scene object model)*

	---

	- ##### 3.3.1. Parenting (Feature)
		***Drag an object onto another to make it a child. The child inherits the parent's transforms — move the parent, the child follows. Unparent to detach and return to independence.***

		*Related:
		- 3.3.2 (hierarchy panel reflects parent/child structure)
		- 1.5.2 (animation tracks — parent and child can both have keyframed animation)
		- 3.2.3 (custom posing — props can be parented to character joints)*

		**Functional requirements:**
		- Any scene element can be parented to any other scene element
		- A child inherits the parent's position, rotation, and scale
			- Moving, rotating, or scaling the parent applies the same transformation to every child (and their children, recursively)
			- The child's own transform is expressed relative to the parent (local space)
		- A scene element can have at most one parent
		- A scene element can have any number of children
		- Parent via viewport drag (drag one object onto another in the 3D viewport) OR via the hierarchy panel (3.3.2). Both methods available
		- Unparenting (unlinking) detaches the child from its parent, making it a root-level element again
		- On unlink: the child stays at its current world-space position, rotation, and scale (the child's local transform is recalculated so it stays exactly where it was visually)
		- On link: the child maintains its current world position by default (no teleport). An option to snap to a parent-relative offset is available but not the default. This accommodates use cases where objects are moved without physical contact (e.g., magic powers, telekinesis).
		- Maximum nesting depth: 4 levels. Attempting to parent beyond 4 levels is prevented
		- Selecting a child and using transform gizmos moves only that child (and its own children), not the parent
		- Deleting a parent deletes the ENTIRE subtree (all children, grandchildren, etc.) — not just the parent
		- Duplicating a parent duplicates the full subtree — no confirmation dialog
		- Relationships are global but DYNAMIC — both between shots AND within a single shot. A prop can be parented to a character in Shot 3 (character picks it up) and independent in Shot 4 (character sets it down). Within a single shot, relationships can change at specific times on the timeline.
		- **Pickwhip linking model:** Relationships within a shot are created using a pickwhip tool (inspired by After Effects). The user drags from one element to another to create a timed link. On the timeline, the user can define when the link starts and when it ends:
			- Example: object is independent at 0:00, pickwhip-linked to a parent at 0:05, unlinked at 0:10. The region between 0:05 and 0:10 is visually greyed out on the child's timeline track, with keyframes on either boundary marking the link/unlink points.
			- While linked (greyed-out region), the child follows the parent. Attempting to keyframe or move the child independently in the greyed-out region has no effect.
			- Outside the linked region, the child is independent and can be keyframed normally.

		> **Note:** Since relationships change per-shot and within shots, it may be more accurate to describe this as positional attachment rather than strict hierarchy. The position of one object (e.g., prop) ties to the position of another (e.g., character's hand).

		**Transform composition:**
		- A child's world transform is its local transform composed with the parent's world transform, evaluated recursively up the chain
		- If both parent and child have animation keyframes in the same shot:
			- The parent's animation is evaluated first
			- The child's animation defines its local-space transform at each point in time
			- The child's world-space result is the composition of parent world transform and child local transform
			- The child's animation does not override or cancel the parent's influence — it layers on top

		**Expected behavior:**
		``` python
			# Basic parenting — table with props
			.if "Plate" is parented to "Table" >>
				||> .if user translates "Table" by (2, 0, 0) >>
					<== "Plate" moves by (2, 0, 0) in world space
					<== "Plate" maintains its position relative to "Table"

			# Rotation inheritance
			.if "Plate" is parented to "Table" >>
				||> .if user rotates "Table" 90 degrees around Y >>
					<== "Plate" orbits around "Table" center, maintaining relative offset
					<== "Plate" rotates 90 degrees around Y (inherits parent rotation)

			# Scale inheritance
			.if "Glass" is parented to "Table" >>
				||> .if user scales "Table" to 2x >>
					<== "Glass" scales to 2x
					<== distance between "Glass" and "Table" center doubles

			# Moving a child independently
			.if "Plate" is parented to "Table" >>
				||> .if user selects "Plate" and translates it >>
					<== "Plate" moves to the new position
					<== "Table" does not move
					<== "Plate" remains a child of "Table"

			# Unparenting preserves world position
			.if "Plate" is parented to "Table" >>
			.if "Table" is at position (5, 0, 3) >>
			.if "Plate" world position is (5, 1, 3) >>
				||> .if user unparents "Plate" >>
					<== "Plate" remains at world position (5, 1, 3)
					<== "Plate" is now a root-level element
					!== "Plate" snaps to its local-space coordinates

			# Recursive inheritance — prop on plate on table
			.if "Fork" is parented to "Plate" >>
			.if "Plate" is parented to "Table" >>
				||> .if user translates "Table" >>
					<== "Plate" follows "Table"
					<== "Fork" follows "Plate" (which follows "Table")

			# Animation composition
			.if "Prop" is parented to "Character" >>
			.if "Character" has position animation in current shot >>
			.if "Prop" has position animation in current shot >>
				||> .if playback is active >>
					<== "Character" evaluates its animation (world space)
					<== "Prop" evaluates its animation (local space, relative to "Character")
					<== "Prop" world position is the composition of both animations
					!== "Prop" animation replaces "Character" influence

			# Parenting via viewport drag
			.if user drags "Plate" onto "Table" in the 3D viewport >>
				<== "Plate" becomes a child of "Table"
				<== "Plate" world position is preserved
				<== hierarchy panel updates to show "Plate" under "Table"

			# Parenting via hierarchy panel drag
			.if user drags "Plate" row onto "Table" row in the hierarchy panel >>
				<== "Plate" becomes a child of "Table"
				<== "Plate" world position is preserved
		```

		**Subtree deletion:**
		``` python
			# Deleting a parent deletes the entire subtree
			.if "Plate" and "Glass" are children of "Table" >>
				||> .if user deletes "Table" >>
					<== "Table" is deleted
					<== "Plate" is deleted
					<== "Glass" is deleted
					!== children become root-level elements
					!== children survive the deletion

			# Deleting a deeply nested parent
			.if "Fork" is parented to "Plate", "Plate" is parented to "Table" >>
				||> .if user deletes "Table" >>
					<== "Table" is deleted
					<== "Plate" is deleted
					<== "Fork" is deleted

			# Deleting a child does not affect the parent
			.if "Plate" is parented to "Table" >>
				||> .if user deletes "Plate" >>
					<== "Plate" is deleted
					<== "Table" remains in the scene
		```

		**Subtree duplication:**
		``` python
			# Duplicating a parent duplicates the full subtree
			.if "Plate" is parented to "Table" >>
				||> .if user selects "Table" and presses Ctrl+D >>
					<== "Table (1)" is created with "Plate (1)" as its child
					<== the duplicate subtree is independent of the original
					<== relative transforms within the duplicate match the original
					!== confirmation dialog shown

			# Duplicating a child
			.if "Plate" is parented to "Table" >>
				||> .if user selects "Plate" and presses Ctrl+D >>
					<== "Plate (1)" is created as a sibling child of "Table"
					<== "Plate (1)" is parented to "Table"
		```

		**Dynamic per-shot relationships:**
		``` python
			# Parenting in one shot, independent in the next
			.if "Sword" is independent in Shot_01 >>
				||> .if user parents "Sword" to "Character" in Shot_02 >>
					<== "Sword" is parented to "Character" in Shot_02
					<== "Sword" remains independent in Shot_01
					<== moving "Character" in Shot_02 moves "Sword" with it
					<== moving "Character" in Shot_01 does NOT move "Sword"

			# Picking up and setting down a prop across shots
			.if "Cup" is independent in Shot_01 >>
			.if user parents "Cup" to "Character" in Shot_02 >>
			.if user unparents "Cup" in Shot_03 >>
				<== Shot_01: "Cup" is independent, sitting on the table
				<== Shot_02: "Cup" is parented to "Character", moves with character
				<== Shot_03: "Cup" is independent again, character set it down

			# Per-shot relationship does not retroactively change earlier shots
			.if "Prop" is independent in Shot_01 through Shot_04 >>
				||> .if user parents "Prop" to "Character" in Shot_03 >>
					<== Shot_01: "Prop" is independent
					<== Shot_02: "Prop" is independent
					<== Shot_03: "Prop" is parented to "Character"
					<== Shot_04: "Prop" is parented to "Character" (relationship carries forward)

			# Changing parent between shots
			.if "Baton" is parented to "Runner_A" in Shot_01 >>
				||> .if user reparents "Baton" to "Runner_B" in Shot_02 >>
					<== Shot_01: "Baton" moves with "Runner_A"
					<== Shot_02: "Baton" moves with "Runner_B"
		```

		**Error cases:**
		``` python
			# Self-parenting
			.if user attempts to parent "Table" to itself >>
				<== operation rejected
				<== no change to hierarchy
				!== circular reference created

			# Circular reference
			.if "B" is parented to "A" >>
				||> .if user attempts to parent "A" to "B" >>
					<== operation rejected
					<== no change to hierarchy

			# Deep circular reference
			.if "C" is parented to "B", "B" is parented to "A" >>
				||> .if user attempts to parent "A" to "C" >>
					<== operation rejected

			# Exceeding max depth
			.if a chain of parents is already 4 levels deep >>
				||> .if user attempts to parent another element as a child of the deepest element >>
					<== operation rejected
					<== visual feedback indicating the nesting limit

			# Reparenting (changing parent)
			.if "Plate" is parented to "Table A" >>
				||> .if user parents "Plate" to "Table B" >>
					<== "Plate" detaches from "Table A"
					<== "Plate" becomes a child of "Table B"
					<== "Plate" world position is preserved during the transfer
		```

		**Interaction with shot continuity (1.4.3):**
		``` python
			# Animating a parent in one shot, creating the next
			.if "Table" (with child "Plate") is animated to position (5, 0, 0) at end of Shot_01 >>
				||> .if user creates Shot_02 >>
					<== Shot_02 initial state has "Table" at (5, 0, 0)
					<== Shot_02 initial state has "Plate" at correct world position relative to "Table"
					<== parent/child relationship carries forward into Shot_02
		```

	- ##### 3.3.2. Hierarchy panel (Feature)
		***A tree view showing every scene object and its parent/child relationships. Click to select, drag to reparent. The director's view of "what goes with what."***

		*Blocked by:
		- 3.3.1 (parenting — the panel visualizes parenting relationships)*

		*Related:
		- 1.3.1 (scene elements — panel lists all scene elements)
		- 1.3.2 (transform gizmos — selecting an element in the panel activates its gizmo)*

		**Functional requirements:**
		- The hierarchy panel is a collapsible sidebar panel
		- Dockable to left or right side of the viewport (user-configurable)
		- The panel displays every scene element as a row in a tree
		- Separate "Lights" section — lights are displayed in their own section of the hierarchy, not mixed with objects
		- Root-level elements (no parent) appear at the top level
		- Children appear indented under their parent
		- Each level of nesting adds one indent step
		- Parent rows have a disclosure triangle (expand/collapse)
			- Collapsed parents hide their children in the panel (objects remain visible in the viewport)
			- Expanded parents show all direct children
			- Default state: all parents expanded
		- Each row displays the element's name
		- The currently selected element is highlighted in the panel
		- Selecting an element in the viewport highlights it in the panel, and vice versa
			- Panel selection and viewport selection are always in sync

		**Interaction:**
		- Clicking a row in the panel selects that element
			- The element is selected in the viewport (selection highlight, gizmo appears)
			- The previously selected element is deselected
		- Clicking empty space in the panel deselects the current element
		- Dragging a row onto another row parents the dragged element to the target
			- A drop indicator shows the target parent during the drag
			- Invalid drops (self-parenting, circular reference, depth exceeded) show a rejection indicator
		- Dragging a row to the root level (top of the panel, or between root-level elements) unparents it
		- The panel updates immediately when elements are added, deleted, duplicated, parented, or unparented

		**Expected behavior:**
		``` python
			# Viewing hierarchy
			.if "Plate" and "Glass" are children of "Table" >>
				<== panel shows "Table" at root level
				<== "Plate" and "Glass" appear indented under "Table"
				<== "Table" row has a disclosure triangle

			# Separate lights section
			.if scene contains "Table", "Chair", "Key Light", "Fill Light" >>
				<== "Table" and "Chair" appear in the objects section
				<== "Key Light" and "Fill Light" appear in a separate "Lights" section
				!== lights mixed in with objects

			# Lights section with parented lights
			.if "Spot Light" is parented to "Camera Rig" >>
				<== "Spot Light" appears under "Camera Rig" in the objects section (parenting takes precedence)
				!== "Spot Light" duplicated in both sections

			# Docking to left side
			.if user configures panel to dock left >>
				<== panel appears on the left side of the viewport
				<== viewport adjusts to fill remaining space

			# Docking to right side
			.if user configures panel to dock right >>
				<== panel appears on the right side of the viewport
				<== viewport adjusts to fill remaining space

			# Selecting via panel
			.if user clicks "Plate" row in the hierarchy panel >>
				<== "Plate" is selected in the viewport
				<== "Plate" row is highlighted in the panel
				<== gizmo appears on "Plate"

			# Selection sync from viewport
			.if user clicks "Glass" in the 3D viewport >>
				<== "Glass" row is highlighted in the panel
				<== panel scrolls to reveal "Glass" if necessary

			# Reparenting via drag
			.if user drags "Lamp" row onto "Table" row >>
				<== "Lamp" becomes a child of "Table"
				<== "Lamp" appears indented under "Table" in the panel
				<== "Lamp" world position is preserved

			# Unparenting via drag to root
			.if "Plate" is a child of "Table" >>
				||> .if user drags "Plate" to root level in the panel >>
					<== "Plate" becomes a root-level element
					<== "Plate" appears at root level in the panel
					<== "Plate" world position is preserved

			# Collapsing a parent
			.if "Table" has children "Plate" and "Glass" >>
				||> .if user clicks the disclosure triangle on "Table" >>
					<== "Plate" and "Glass" rows hidden in the panel
					<== "Plate" and "Glass" remain visible in the viewport
					!== children hidden from the 3D view

			# Invalid drop feedback
			.if "B" is parented to "A" >>
				||> .if user drags "A" onto "B" >>
					<== drop indicator shows rejection (color change or icon)
					<== releasing the drag does nothing
					!== circular reference created

			# Element added to scene
			.if user imports or duplicates an element >>
				<== new element appears in the panel immediately
				<== new element appears at root level (unless duplicated from a child — see 3.3.1)

			# Element deleted from scene (subtree deletion)
			.if user deletes "Table" (which has children "Plate" and "Glass") >>
				<== "Table" row removed from panel
				<== "Plate" row removed from panel
				<== "Glass" row removed from panel

			# Panel reflects per-shot relationships
			.if "Sword" is independent in Shot_01 but parented to "Character" in Shot_02 >>
				||> .if user is viewing Shot_01 >>
					<== "Sword" appears at root level in the panel
				||> .if user switches to Shot_02 >>
					<== "Sword" appears indented under "Character" in the panel
		```

		**Error cases:**
		``` python
			# Empty scene
			.if no scene elements exist >>
				<== panel shows empty state (no rows)
				!== panel crashes or shows stale data

			# Deeply nested tree
			.if hierarchy is 4 levels deep >>
				<== all 4 levels visible when expanded
				<== indentation remains legible at every level
				<== dragging onto the deepest element shows rejection indicator
		```
