# Milestone 3.3: Multi-camera

**Date**: 2026-03-10
**Status**: Draft
**Blocked by**: 1.5 (Keyframe animation -- cameras have animated paths), 3.2 (Characters -- multi-cam is most useful with actors to film)

---

- ### 3.3. Multi-camera (Milestone)

  Multi-camera is how real production works. A director doesn't shoot a scene once from one angle -- they plan coverage. The wide master, the over-the-shoulder, the close-up reaction. Each camera setup has its own position, lens, and movement. The director needs to see how they cut together before the crew shows up.

  This milestone builds in layers, the way a director's needs escalate. First: bookmarks -- saved framings the director can flip between while blocking a scene. This is lightweight and immediately useful. Then: independent cameras that live in the scene, each with their own lens and animation. Then: assigning those cameras to shots, so the sequence becomes a multi-camera edit. Finally: the video village experience -- seeing multiple angles simultaneously to evaluate coverage.

  The single-camera model from Milestone 1.1 remains the default. Multi-camera is additive. A project with one camera works exactly as it always did. Multi-camera features appear only when the director creates additional cameras or bookmarks.

  - ##### 3.3.1. Camera bookmarks (Feature)

    ***Save the current camera position, rotation, and focal length as a named bookmark. Recall it instantly. A director tries three framings for a scene and wants to flip between them without losing any.***

    *Related:
    - 1.1.1 (camera movement -- bookmarks capture the result of camera positioning)
    - 1.1.2 (lens system -- bookmarks include focal length)*

    **Functional requirements:**
    - A bookmark captures three values: position (world-space), rotation (world-space), and focal length
    - Bookmarks are named. Default name follows the pattern "Bookmark_NN" (zero-padded sequential: Bookmark_01, Bookmark_02, etc.)
    - The user can rename a bookmark to any non-empty string up to 32 characters
    - Bookmarks are ordered by creation time
    - The user can create a bookmark from the current camera state at any time
    - Recalling a bookmark sets the camera to the bookmarked position, rotation, and focal length
    - Recall is instant -- no animated transition. The director is comparing framings, not watching a dolly move
    - Bookmarks are scene-level data -- they are not tied to any particular shot
    - Creating a bookmark does not alter the current shot's camera animation or keyframes
    - Recalling a bookmark does not alter the current shot's camera animation or keyframes -- it moves the camera to the bookmarked state for viewing purposes only
    - Bookmarks are independent of each other. Modifying one does not affect others
    - The user can update an existing bookmark to capture the current camera state (overwrite)
    - The user can delete a bookmark
    - There is no limit on the number of bookmarks

    **Design constraints:**
    - Bookmarks are distinct from cameras (3.3.2). A bookmark is a saved transform -- it has no animation, no independent existence in the scene, no visual representation in the viewport. It is a recall point, not an entity.
    - Bookmarks are accessible from a list or panel. They must be reachable within one click -- the value is speed of comparison.

    **Expected behavior:**
    ``` python
      # creating a bookmark
      .if camera is at position P using rotation R and focal length 35mm
      ||> .if user creates a bookmark >>
          <== bookmark "Bookmark_01" is created
          <== bookmark stores P, R, and 35mm
          <== camera position is unchanged

      # recalling a bookmark
      .if camera is currently at a different position using 85mm
      ||> .if user recalls "Bookmark_01" >>
          <== camera instantly jumps to position P using rotation R and 35mm
          !== camera animates or transitions to the bookmarked state

      # bookmark does not affect shot animation
      .if current shot has a keyframed camera animation
      ||> .if user recalls a bookmark >>
          <== camera moves to bookmark state
          <== the shot's camera keyframes are not modified
          <== if user presses play, animation plays from its original keyframes
          !== bookmark recall inserts or overwrites keyframes

      # updating a bookmark
      .if "Bookmark_01" exists using position P1
      ||> .if user moves camera to position P2
      ||> .if user updates "Bookmark_01" >>
          <== "Bookmark_01" now stores P2 and the current rotation and focal length
          <== previous values are overwritten

      # A/B comparison workflow
      .if user creates "Bookmark_01" using wide framing at 24mm
      ||> .if user moves camera and creates "Bookmark_02" using tight framing at 85mm
      ||> .if user recalls "Bookmark_01" >>
          <== camera instantly shows the wide framing
      ||> .if user recalls "Bookmark_02" >>
          <== camera instantly shows the tight framing
          <== director can flip back and forth without delay

      # renaming
      .if user double-clicks "Bookmark_01" name >>
          <== name becomes editable
          <== accepts any non-empty string up to 32 characters

      .if user clears name field and confirms >>
          <== name reverts to previous value
          !== empty string accepted

      # deletion
      .if user deletes "Bookmark_02" >>
          <== "Bookmark_02" is removed from the bookmark list
          <== "Bookmark_01" is unaffected
          <== camera position is unchanged
    ```

    **Error cases:**
    ``` python
      # recall with no bookmarks
      .if no bookmarks exist >>
          <== bookmark list is empty
          <== no recall action available
          !== error or crash

      # bookmark after camera reset
      .if user resets camera to defaults
      ||> .if user creates a bookmark >>
          <== bookmark captures the default camera state
          <== bookmark functions normally
    ```

  - ##### 3.3.2. Camera creation (Feature)

    ***Add independent cameras to the scene. Each camera has its own name, position, lens, and animation. Camera A is the master wide. Camera B is the OTS. Camera C is the CU. They exist simultaneously in the 3D world.***

    *Blocked by:
    - 3.3.1 (camera bookmarks -- bookmarks establish the concept of named camera states; cameras extend this to persistent scene entities)*

    *Related:
    - 1.1.1 (camera movement -- each camera has the same movement capabilities)
    - 1.1.2 (lens system -- each camera has its own focal length)
    - 1.5 (keyframe animation -- each camera has independent animation tracks)*

    **Functional requirements:**
    - The project starts with one camera: "Camera_A". This is the original single camera from Milestone 1.1, now named
    - The user can add cameras. New cameras are named sequentially: Camera_B, Camera_C, Camera_D, etc.
    - The user can rename any camera to any non-empty string up to 32 characters
    - Each camera independently stores: position, rotation, focal length, camera body preset, lens set preset, depth of field settings, and its own keyframed animation per shot
    - The user can switch which camera they are "viewing through" -- the active camera
    - The active camera is the one whose perspective fills the viewport. All camera movement inputs (pan, tilt, dolly, etc.) affect the active camera
    - Cameras that are not active can optionally be displayed in the 3D viewport as camera icons or frustum wireframes, so the director can see where all cameras are placed
    - Non-active camera representations are scene elements -- they can be selected and repositioned using transform gizmos, just like any other object
    - Repositioning a non-active camera via gizmo updates that camera's position. If the director views through that camera later, they see the updated framing
    - The user can delete any camera except the last remaining one. At least one camera must always exist
    - Each camera maintains its own independent animation data per shot. Camera_A's animation in Shot 1 is completely separate from Camera_B's animation in Shot 1

    **Design constraints:**
    - Switching the active camera is a viewport operation, not a scene change. The scene does not change -- only the lens the director is looking through
    - Camera representations in the viewport (frustum wireframes, camera icons) must be visually distinct from scene objects so the director does not confuse a camera marker with a prop

    **Expected behavior:**
    ``` python
      # initial state
      .if project is opened for the first time >>
          <== one camera exists: "Camera_A"
          <== Camera_A is the active camera
          <== no other cameras visible

      # adding a camera
      .if user adds a camera >>
          <== "Camera_B" is created
          <== Camera_B's initial position is the current viewport camera position
          <== Camera_B's initial focal length matches the current focal length
          <== Camera_B becomes the active camera
          <== Camera_A's frustum wireframe appears in the 3D viewport

      # switching active camera
      .if Camera_A is active
      ||> .if user switches to Camera_B >>
          <== viewport shows Camera_B's perspective
          <== camera movement inputs now affect Camera_B
          <== Camera_A's frustum wireframe appears in the viewport
          <== Camera_B's frustum wireframe disappears (you do not see yourself)

      # independent lens settings
      .if Camera_A uses 24mm on ARRI Alexa 35
      .if Camera_B uses 85mm on RED V-Raptor >>
          <== switching to Camera_A shows 24mm FOV using Alexa 35 sensor
          <== switching to Camera_B shows 85mm FOV using V-Raptor sensor
          <== HUD updates to reflect the active camera's settings

      # repositioning a non-active camera via gizmo
      .if Camera_A is active
      .if user selects Camera_B's frustum wireframe in the viewport
      ||> .if user drags Camera_B using the translate gizmo >>
          <== Camera_B's position updates
          <== Camera_A's view is unchanged (the director is still looking through Camera_A)
      ||> .if user switches to Camera_B >>
          <== viewport shows the new position that was set via gizmo

      # independent animation per shot
      .if Shot 1 is active
      .if user animates Camera_A along a dolly track >>
          <== Camera_A has keyframes in Shot 1
          <== Camera_B has no keyframes in Shot 1 (unless separately animated)
          <== Camera_A's keyframes in Shot 2 are unaffected

      # renaming
      .if user renames "Camera_B" to "OTS" >>
          <== camera is now labeled "OTS" everywhere it appears
          <== functionality is unchanged

      # deletion
      .if three cameras exist: Camera_A, Camera_B, Camera_C
      ||> .if user deletes Camera_B >>
          <== Camera_B is removed from the scene
          <== Camera_B's frustum wireframe disappears
          <== if Camera_B was the active camera, Camera_A becomes active
          <== Camera_A and Camera_C are unaffected

      # cannot delete last camera
      .if only Camera_A exists
      ||> .if user attempts to delete Camera_A >>
          <== deletion is prevented
          !== scene left with zero cameras
    ```

    **Error cases:**
    ``` python
      # adding a camera with no shot active
      .if sequence is empty (no shots)
      ||> .if user adds a camera >>
          <== camera is created using current viewport state
          <== camera has no animation data (no shots to animate within)
          <== camera functions normally once shots are created

      # all cameras at the same position
      .if Camera_A and Camera_B occupy the same position >>
          <== both cameras function independently
          <== frustum wireframes may overlap visually but remain individually selectable
          !== cameras merge or conflict
    ```

  - ##### 3.3.3. Camera-to-shot assignment (Feature)

    ***Assign a camera to each shot. "Shot 1 uses Camera A, Shot 2 uses Camera B, Shot 3 uses Camera A again." This is how a director plans coverage -- deciding which angle covers which moment.***

    *Blocked by:
    - 3.3.2 (camera creation -- must have multiple cameras to assign)
    - 1.4 (shot sequencer -- must have shots to assign cameras to)*

    *Related:
    - 1.5 (keyframe animation -- each camera-shot pairing has its own animation)*

    **Functional requirements:**
    - Every shot is assigned to exactly one camera
    - By default, every shot is assigned to Camera_A (backward compatible with the single-camera model)
    - The user can reassign any shot to any camera in the scene
    - Multiple shots can be assigned to the same camera
    - When a shot is selected in the sequencer, the viewport switches to that shot's assigned camera
    - The sequencer thumbnail for each shot renders from that shot's assigned camera perspective
    - The shot's assigned camera name is displayed on the shot card in the sequencer
    - Each camera-shot pairing has independent animation data. Camera_A in Shot 1 can have a dolly move, while Camera_A in Shot 3 can have a static locked-off framing -- these are separate animation tracks
    - Reassigning a shot from Camera_A to Camera_B does not transfer animation data. The shot begins using Camera_B's current state with no keyframes (a single keyframe at t=0 capturing Camera_B's current position, rotation, and focal length)
    - Sequence playback cuts between cameras as it advances through shots. If Shot 1 is Camera_A and Shot 2 is Camera_B, playback switches the viewport from Camera_A to Camera_B at the cut point

    **Design constraints:**
    - Camera assignment is a shot-level property, not a sequence-level property. Each shot independently declares which camera it uses
    - The assignment UI must be lightweight -- a dropdown or single-click selection on the shot card. Assigning cameras should be as fast as naming shots

    **Expected behavior:**
    ``` python
      # default assignment
      .if user creates three shots before creating any additional cameras >>
          <== Shot_01, Shot_02, Shot_03 are all assigned to Camera_A
          <== sequencer cards show "Camera_A" on each shot

      # reassigning a shot
      .if Camera_A and Camera_B exist
      ||> .if user assigns Shot_02 to Camera_B >>
          <== Shot_02's card shows "Camera_B"
          <== Shot_02's thumbnail re-renders from Camera_B's perspective
          <== selecting Shot_02 in the sequencer switches the viewport to Camera_B

      # same camera, multiple shots, independent animation
      .if Shot_01 and Shot_03 are both assigned to Camera_A >>
          <== Shot_01 has its own animation data for Camera_A
          <== Shot_03 has its own animation data for Camera_A
          <== modifying Camera_A's animation in Shot_01 does not affect Shot_03

      # reassignment clears animation
      .if Shot_02 is assigned to Camera_A using a keyframed dolly animation
      ||> .if user reassigns Shot_02 to Camera_B >>
          <== Shot_02's camera animation is reset to a single keyframe at t=0 capturing Camera_B's current state
          <== the previous Camera_A animation data for Shot_02 is discarded
          !== Camera_A's animation data transferred to Camera_B

      # sequence playback across cameras
      .if Shot_01 uses Camera_A, Shot_02 uses Camera_B, Shot_03 uses Camera_A
      ||> .if user plays the sequence from Shot_01 >>
          <== Shot_01 plays through Camera_A
          <== at the cut to Shot_02, viewport switches to Camera_B
          <== at the cut to Shot_03, viewport switches back to Camera_A
          <== transitions are instantaneous cuts -- no blending between cameras

      # selecting a shot switches camera
      .if Shot_01 (Camera_A) is current
      ||> .if user clicks Shot_02 (Camera_B) in the sequencer >>
          <== viewport switches to Camera_B's perspective at Shot_02's t=0
          <== camera movement inputs now affect Camera_B
          <== HUD updates to show Camera_B's settings (focal length, body, lens set)

      # shot thumbnail reflects assigned camera
      .if Shot_01 is assigned to Camera_A framing a wide shot
      .if Shot_02 is assigned to Camera_B framing a close-up >>
          <== Shot_01's thumbnail shows the wide framing
          <== Shot_02's thumbnail shows the close-up framing
    ```

    **Error cases:**
    ``` python
      # deleting a camera that shots are assigned to
      .if Camera_B has shots assigned to it (Shot_02, Shot_04)
      ||> .if user deletes Camera_B >>
          <== Shot_02 and Shot_04 are reassigned to Camera_A (the first remaining camera)
          <== Shot_02 and Shot_04's camera animation is reset to a single keyframe at t=0 capturing Camera_A's current state
          <== sequencer thumbnails for Shot_02 and Shot_04 re-render from Camera_A
          !== shots left assigned to a deleted camera
          !== application error or orphaned shots

      # reassigning to the same camera
      .if Shot_01 is assigned to Camera_A
      ||> .if user assigns Shot_01 to Camera_A >>
          <== nothing changes
          <== animation data preserved
          !== animation data reset

      # creating a shot when multiple cameras exist
      .if Camera_A and Camera_B exist
      .if Camera_B is the active camera
      ||> .if user creates a new shot >>
          <== new shot is assigned to Camera_B (the currently active camera)
          !== new shot always defaults to Camera_A
    ```

  - ##### 3.3.4. Multi-cam preview (Feature)

    ***See multiple camera angles at once. Split the viewport into two or four panels, or overlay a picture-in-picture window. The video village experience -- the director evaluates coverage without clicking between shots.***

    *Blocked by:
    - 3.3.2 (camera creation -- must have multiple cameras to preview simultaneously)
    - 3.3.3 (camera-to-shot assignment -- preview is most useful when cameras are assigned to shots)*

    **Functional requirements:**
    - Multi-cam preview divides the viewport to show multiple camera feeds simultaneously
    - Available layouts:
      - Single (default) -- full viewport, one camera. This is the standard mode
      - Side-by-side -- two cameras, split vertically (left/right)
      - Quad -- four cameras in a 2x2 grid
      - Picture-in-picture -- one camera fills the viewport, a second camera appears in a smaller overlay window in a corner
    - Each panel in a multi-cam layout shows a live render from its assigned camera
    - One panel is always designated as the "primary" panel. The primary panel has a visible border or indicator
    - Camera movement inputs (pan, tilt, dolly, etc.) affect only the primary panel's camera
    - The user can click a panel to make it the primary panel
    - Each panel displays that camera's name
    - Each panel respects the active aspect ratio masks and frame guides
    - Multi-cam preview is a viewport mode -- it does not alter shot assignments, camera data, or the sequence
    - Panels render in real time. If the active shot is playing back, every panel updates every frame from its respective camera
    - The user can assign which camera appears in each panel
    - In picture-in-picture mode, the PiP window can be positioned in any corner of the viewport
    - The user can resize the PiP window within reasonable bounds (not so small it is useless, not so large it occludes the main view)
    - Exiting multi-cam preview returns to single-camera mode showing the primary panel's camera

    **Design constraints:**
    - Multi-cam preview must maintain interactive frame rates. Rendering two or four camera views is more expensive than one. If frame rates drop below usable thresholds, the preview loses its value
    - The director should be able to enter and exit multi-cam preview quickly -- it is a comparison mode, not a permanent workflow change
    - Panel borders must be visually clear so the director knows which camera they are looking through in each panel

    **Expected behavior:**
    ``` python
      # entering side-by-side mode
      .if Camera_A and Camera_B exist
      ||> .if user activates side-by-side preview >>
          <== viewport splits into two panels
          <== left panel shows Camera_A's view
          <== right panel shows Camera_B's view
          <== each panel displays the camera name
          <== one panel is highlighted as primary

      # switching primary panel
      .if side-by-side mode is active using Camera_A as primary (left panel)
      ||> .if user clicks the right panel (Camera_B) >>
          <== right panel becomes primary
          <== camera movement inputs now affect Camera_B
          <== primary indicator moves to the right panel

      # quad view
      .if four cameras exist: Camera_A, Camera_B, Camera_C, Camera_D
      ||> .if user activates quad preview >>
          <== viewport splits into 2x2 grid
          <== each panel shows one camera
          <== user can click any panel to make it primary

      # picture-in-picture
      .if Camera_A and Camera_B exist
      ||> .if user activates PiP preview >>
          <== Camera_A fills the main viewport
          <== Camera_B appears in a small overlay window in one corner
          <== user can drag PiP window to any corner
          <== user can resize PiP window within min/max bounds

      # live playback across panels
      .if side-by-side mode is active
      ||> .if user plays the current shot >>
          <== both panels update every frame
          <== Camera_A's animation plays in its panel
          <== Camera_B's animation plays in its panel (if Camera_B has animation in this shot)
          <== playback is synchronized -- both panels show the same timecode

      # aspect ratio masks in panels
      .if aspect ratio is set to 2.39:1
      .if side-by-side mode is active >>
          <== both panels display letterbox bars appropriate to their dimensions
          <== frame guides appear within the unmasked area of each panel

      # exiting multi-cam preview
      .if side-by-side mode is active using Camera_B as primary
      ||> .if user exits multi-cam preview >>
          <== viewport returns to single full-viewport mode
          <== viewport shows Camera_B (the primary panel's camera)

      # reassigning panel cameras
      .if side-by-side mode shows Camera_A (left) and Camera_B (right)
      ||> .if user assigns Camera_C to the right panel >>
          <== right panel now shows Camera_C's view
          <== left panel unchanged
          <== Camera_B is not affected (still exists, just not shown in this layout)

      # fewer cameras than panels
      .if only two cameras exist
      ||> .if user activates quad preview >>
          <== two panels show Camera_A and Camera_B
          <== remaining two panels show a neutral state (empty or repeated camera)
          !== application error
    ```

    **Error cases:**
    ``` python
      # multi-cam with one camera
      .if only Camera_A exists
      ||> .if user activates side-by-side preview >>
          <== both panels show Camera_A
          <== preview functions but provides no comparative value
          !== feature blocked or error shown

      # deleting a camera shown in a panel
      .if side-by-side mode shows Camera_A and Camera_B
      ||> .if user deletes Camera_B >>
          <== Camera_B's panel switches to Camera_A (or the next available camera)
          !== empty panel or crash

      # PiP minimum size
      .if user drags PiP window smaller and smaller >>
          <== PiP window stops shrinking at a minimum usable size
          !== PiP window shrinks to zero pixels

      # PiP maximum size
      .if user drags PiP window larger and larger >>
          <== PiP window stops growing before it occludes the majority of the main view
          !== PiP window fills the entire viewport
    ```

## Acceptance Criteria

### Camera Bookmarks

1. User positions camera at a wide shot (24mm), creates bookmark "Wide"
   - <== "Wide" appears in the bookmark list
   - <== camera position unchanged

2. User repositions camera to a close-up (85mm), creates bookmark "CU"
   - <== "CU" appears in the bookmark list

3. User recalls "Wide"
   - <== camera instantly shows the 24mm wide framing
   - <== no animation or transition

4. User recalls "CU"
   - <== camera instantly shows the 85mm close-up

5. User updates "Wide" to a new position
   - <== recalling "Wide" shows the new position, not the original

6. User deletes "CU"
   - <== "CU" removed from list
   - <== "Wide" still works

### Camera Creation and Management

7. User adds Camera_B
   - <== two cameras exist: Camera_A, Camera_B
   - <== Camera_B is active
   - <== Camera_A's frustum wireframe visible in the viewport

8. User switches active camera to Camera_A
   - <== viewport shows Camera_A's perspective
   - <== Camera_B's frustum wireframe visible

9. User sets Camera_A to 24mm and Camera_B to 85mm, then switches between them
   - <== HUD updates to show the correct focal length for each
   - <== FOV changes appropriately

10. User selects Camera_B's frustum wireframe and moves it via translate gizmo
    - <== Camera_B's position updates
    - <== switching to Camera_B shows the new framing

### Camera-to-Shot Assignment

11. User creates three shots and two cameras. Assigns Shot_01 to Camera_A, Shot_02 to Camera_B, Shot_03 to Camera_A
    - <== sequencer cards show the correct camera names
    - <== clicking each shot switches to the assigned camera

12. User plays the full sequence
    - <== playback cuts between Camera_A and Camera_B at shot boundaries
    - <== each shot plays its own camera animation

13. User animates Camera_A with a dolly move in Shot_01. Camera_A in Shot_03 remains static
    - <== Shot_01 plays the dolly animation
    - <== Shot_03 shows Camera_A in a locked-off position

14. User reassigns Shot_02 from Camera_B to Camera_A
    - <== Shot_02's animation resets to Camera_A's current state at t=0
    - <== Shot_02's thumbnail re-renders from Camera_A

15. User deletes Camera_B while Shot_04 is assigned to it
    - <== Shot_04 reassigns to Camera_A
    - <== Shot_04's animation resets

### Multi-cam Preview

16. User enters side-by-side mode with Camera_A and Camera_B
    - <== viewport splits into two panels showing both cameras
    - <== clicking a panel makes it primary

17. User plays back a shot while in side-by-side mode
    - <== both panels update in sync every frame

18. User enters PiP mode
    - <== main viewport shows one camera, small overlay shows another
    - <== PiP window can be repositioned to any corner

19. User exits multi-cam preview
    - <== viewport returns to single-camera mode showing the primary camera

### Edge Cases

20. User recalls a bookmark while a shot is playing
    - <== playback stops
    - <== camera jumps to bookmarked state
    - <== shot's camera keyframes are not modified

21. User creates a bookmark, then creates Shot_02 and recalls the bookmark while Shot_02 is active
    - <== camera shows bookmarked state
    - <== Shot_02's animation is not altered
    - <== pressing play resumes Shot_02's original animation

22. Two shots use Camera_A. User edits Camera_A's animation in Shot_01
    - <== Shot_03's Camera_A animation is completely unaffected

23. User adds Camera_C while in multi-cam preview
    - <== Camera_C can be assigned to any panel
    - <== preview continues functioning
