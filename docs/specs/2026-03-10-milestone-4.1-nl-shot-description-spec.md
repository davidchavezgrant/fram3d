- ### 4.1. Natural language shot description (Milestone)

  A director walks onto set and tells the DP: "Let's start with a medium close-up, slightly low angle, camera left." The DP knows exactly what that means because cinematographers share a precise vocabulary -- shot sizes, angles, and movement types that have meant the same thing for a hundred years. This milestone teaches Fram3d that vocabulary so a director can type a shot description and get a camera position as a starting point.

  The emphasis is on *starting point*. The system will not nail the perfect shot on the first try. What it should do is get close enough that the director spends thirty seconds refining instead of two minutes manually positioning the camera from scratch. If the system produces a medium close-up when the director asked for a medium close-up, it has done its job -- even if the exact framing needs adjustment.

  This is explicitly speculative. Sequencing principle #4 applies: "AI is upside, not the product." Fram3d must be a great previsualization tool without this feature. Natural language shot description is a workflow accelerator for users who want it, not a core dependency. If it does not work well enough to be useful, it ships as an experimental feature or not at all.

  **What might not work well:**
  - Subject resolution in complex scenes. "Close-up of the actor on the left" requires the system to determine which actor is "on the left" relative to the camera or the scene. Spatial reasoning from natural language is hard and may produce wrong results frequently enough to frustrate users.
  - Ambiguous or contradictory descriptions. "A tight wide shot" is contradictory. "A dramatic angle" is subjective. The system will encounter descriptions it cannot meaningfully interpret, and it needs to fail gracefully rather than produce nonsense.
  - Shots that reference objects or characters not present in the scene. "Close-up of the knife on the counter" when there is no knife and no counter. The system must recognize what it cannot do.
  - The gap between cinematic intent and geometric output. "An intimate two-shot" carries emotional meaning that is hard to translate into camera coordinates. The system may technically satisfy the shot size while missing the feel entirely.

  *Related: 1.1.1 Camera movement, 1.1.2 Lens system, 1.1.4 Focus, 3.2.1 Mannequin placement*

## 4.1.1 Shot-from-text

***Type a shot description in plain language -- "medium close-up of the actor at the table, slightly low angle" -- and the camera repositions to match. Not the final shot, but a starting point the director refines manually.***

### Input

- The user enters a free-text shot description via the command palette (on-demand, not a persistent text field)
- Input is plain language, not a structured form or dropdown selection
- Descriptions may include any combination of: shot size, camera angle, subject reference, spatial positioning, lens preference, movement instruction
- Descriptions may be partial -- "close-up" with no subject specified is valid
- Compound descriptions create multiple shots. "OTS then reverse" creates two shots in the sequencer.
- Input language is English only for initial release

### Interpretation

- The system parses the description and extracts actionable parameters:
  - **Shot size**: mapped from the shot vocabulary (4.1.2) to a subject-to-camera distance and focal length
  - **Camera angle**: mapped to a vertical offset and tilt relative to the subject
  - **Subject**: resolved to a specific scene object or character, or to the scene as a whole
  - **Spatial modifiers**: "camera left," "camera right," "from behind," "profile" -- mapped to a horizontal angle around the subject
  - **Lens cues**: descriptions that imply focal length ("compressed background," "wide and distorted") influence lens selection beyond what the shot size implies

- Interacts with auto-keyframing: when the AI repositions the camera, it creates keyframes via the standard auto-keyframing system. The shot-from-text action is not a special case -- it moves the camera, and auto-keyframing captures that movement like any other camera change.

- Stores movement metadata ("slow dolly in," "tracking shot") for potential future use by the animation system, even if movement generation is not implemented yet. This metadata is attached to the shot in the sequencer so it is available when movement generation ships.

- The system does NOT interpret:
  - Camera movement or animation ("dolly in slowly"). This milestone positions the camera; it does not animate it. Movement descriptions are acknowledged and stored as metadata for future milestones.
  - Lighting descriptions ("dark and moody"). Lighting is a separate system.
  - Editing instructions ("cut to," "dissolve to"). These are sequencer-level operations.

### Subject Resolution

``` python
  .if >> description names a specific character ("the woman," "Actor_02," a character's given name)
      <== system attempts to match the reference to a character in the scene
      .if >> exactly one character matches
          <== that character is used as the subject
      .if >> multiple characters could match
          <== system uses the best match based on available information (name, position)
          <== if still ambiguous, system notifies the user which subject it chose
      .if >> no character matches the reference
          <== system notifies the user that the referenced subject was not found
          <== camera does not reposition

  .if >> description names a non-character object ("the table," "the car," "the doorway")
      <== system attempts to match the reference to a scene object by name
      <== matching rules follow the same pattern as character resolution

  .if >> description uses spatial references ("the actor on the left," "the person closest to the window")
      <== system evaluates spatial relationships relative to the current camera position
      <== "left" and "right" are interpreted from the camera's perspective, not the scene's
      <== this is the hardest resolution problem and may frequently produce incorrect results

  .if >> description references no subject ("establishing wide," "high angle of the room")
      <== system frames the entire scene or visible area
      <== camera positions to see the full extent of scene objects

  .if >> only one character exists in the scene and the description says "close-up" with no subject
      <== system assumes the single character is the subject

  .if >> multiple characters exist and the description specifies no subject
      <== system frames all visible characters as a group
      !== system randomly picks one character
```

### Camera Positioning

The system computes a target camera state: position, rotation, and focal length.

``` python
  .if >> description specifies a shot size and a subject
      <== camera positions at a distance from the subject that produces the described framing
      <== focal length is set to a value appropriate for the shot size (wider for wide shots, longer for close-ups)
      <== both distance and focal length work together -- the system does not simply zoom from the current position

  .if >> description specifies a camera angle
      <== camera height adjusts relative to the subject to produce the described angle
      <== camera tilts to keep the subject in frame

  .if >> description specifies spatial positioning ("camera left," "three-quarter angle")
      <== camera orbits around the subject to the described position
      <== camera continues to face the subject

  .if >> description specifies nothing beyond a shot size
      <== camera maintains its current horizontal angle relative to the subject
      <== only distance and height adjust
```

### Transition

``` python
  .if >> system successfully interprets the description
      <== camera transitions smoothly from its current position to the computed position
      <== transition uses the same smooth interpolation as the focus system (1.1.4)
      !== camera teleports instantly to the new position

  .if >> user interrupts the transition with manual camera input
      <== transition is cancelled immediately
      <== user's manual input takes effect
      <== the partially-completed position is where the camera stays
```

### Feedback

``` python
  .if >> system successfully positions the camera
      <== a brief confirmation is displayed showing how the system interpreted the description (e.g., "MCU, low angle, camera left of Actor_01")
      <== this interpretation summary lets the user verify the system understood them

  .if >> system cannot interpret the description
      <== a message explains what was not understood
      <== camera does not move
      !== system silently fails or produces a random camera position

  .if >> system interprets the description but with low confidence
      <== system proceeds with its best interpretation
      <== feedback indicates uncertainty (e.g., "Interpreted as medium shot -- subject unclear, framing all characters")
```

### Focal Length Selection

Shot descriptions carry implicit lens information. The system should set focal length, not just camera position.

- Close-ups and tighter imply longer lenses. An ECU at 24mm looks wrong -- the face distorts. The system should select a focal length that a cinematographer would naturally reach for.
- Wide establishing shots imply wider lenses. An ELS at 135mm would require the camera to be implausibly far from the scene.
- When the description explicitly references a lens quality ("compressed," "distorted," "flat"), the system biases its focal length selection accordingly.
- The selected focal length works within the constraints of the current camera body's sensor size and the available lens set. The system should snap to the nearest available prime in the current lens set when possible.

``` python
  # shot size implies focal length
  .if description is "extreme close-up" with subject >>
      <== system selects a longer focal length (biased toward 85mm-135mm range)
      <== camera positions close to subject
      !== system uses a wide lens and moves extremely close, producing distortion

  .if description is "extreme long shot" >>
      <== system selects a wider focal length (biased toward 18mm-35mm range)
      <== camera positions far from subjects to capture the full environment

  # explicit lens cue overrides default
  .if description is "close-up, wide lens" >>
      <== system uses a wider focal length than it normally would for a close-up
      <== this is a deliberate creative choice -- the user wants the distortion
```

### Non-Subject Shots

Not every shot references a person or object. The system must handle compositional and environmental descriptions.

``` python
  .if >> description is a scene-level shot ("establishing wide of the room," "overhead of the table")
      <== camera positions to show the described scope of the environment
      <== framing encompasses visible scene objects without targeting any single one

  .if >> description references a spatial location rather than an object ("looking down the hallway," "from the doorway")
      <== system interprets spatial locations as best it can from scene geometry
      <== this will often fail or produce approximate results -- spatial location parsing is extremely difficult without explicit scene markup
```

### Error Handling

``` python
  # contradictory description
  .if description contains contradictory terms ("tight wide shot," "low angle bird's eye") >>
      <== system identifies the contradiction
      <== system uses the first meaningful term and ignores the contradictory one
      <== feedback explains what was used and what was discarded

  # description references nonexistent subject
  .if description says "close-up of the dog" and no dog exists within the scene >>
      <== system reports that no matching subject was found
      <== camera does not reposition
      !== system frames a random object and calls it "the dog"

  # empty or nonsensical input
  .if description is empty, random characters, or contains no recognizable cinematic terms >>
      <== system reports that it could not interpret the input
      <== camera does not reposition

  # description is only a movement instruction
  .if description is "slow dolly in" with no shot size, angle, or subject >>
      <== system reports that movement instructions are not yet supported
      <== camera does not reposition
      <== the movement description may be stored as metadata for future use
```

### Compound Descriptions

Compound descriptions that describe a sequence of shots create multiple shots in the sequencer rather than a single camera position.

``` python
  .if >> description contains a sequence of shots ("OTS then reverse," "wide establishing then cut to close-up")
      <== system creates one shot per segment in the sequencer
      <== each shot is independently positioned using the same interpretation pipeline
      <== shots are ordered in the sequencer in the order they appear in the description
      !== system combines the segments into a single averaged camera position

  .if >> description is "OTS from Actor_01 to Actor_02 then reverse"
      <== system creates two shots in the sequencer
      <== first shot: OTS from behind Actor_01's shoulder facing Actor_02
      <== second shot: OTS from behind Actor_02's shoulder facing Actor_01
      <== both shots are created as distinct entries in the sequencer

  .if >> compound description contains one segment the system cannot interpret
      <== system creates shots for the segments it can interpret
      <== feedback identifies which segment failed and why
      !== system discards all segments because one failed
```

### Auto-Keyframe Interaction

Shot-from-text repositions the camera through the standard camera control path, so auto-keyframing applies naturally.

``` python
  .if >> auto-keyframing is enabled and system repositions the camera from a shot description
      <== the resulting camera position is captured as keyframes by the auto-keyframing system
      <== no special integration is needed -- the camera moved, auto-keyframing records it
      !== shot-from-text bypasses the auto-keyframing system

  .if >> auto-keyframing is disabled and system repositions the camera from a shot description
      <== camera repositions but no keyframes are created
      <== behavior is identical to manually moving the camera with auto-keyframing off

  .if >> compound description creates multiple shots with auto-keyframing enabled
      <== each shot's camera position generates its own keyframes in the sequencer
      <== the keyframes correspond to the individual shots, not to the transition between them
```

## 4.1.2 Shot Vocabulary

***The system understands the standardized language of cinematography -- the same vocabulary taught in film schools and used on professional sets worldwide. Shot sizes, camera angles, and movement types each map to concrete camera parameters.***

### Shot Sizes

The system recognizes the following shot sizes. Each maps to a framing ratio -- how much of the subject (when the subject is a person) fills the vertical frame.

| Abbreviation | Full name | Framing (human subject) |
|---|---|---|
| ECU | Extreme close-up | A single feature: eyes, mouth, hands. The subject fills or overflows the frame. |
| CU | Close-up | Head and shoulders. Face fills most of the frame. |
| MCU | Medium close-up | Head to mid-chest. The standard "talking head" framing. |
| MS | Medium shot | Head to waist. Subject and immediate environment visible. |
| MLS | Medium long shot | Head to knees. Also called the "cowboy shot" -- enough to see a holster. |
| LS | Long shot | Full body, head to feet, with some environment above and below. |
| ELS | Extreme long shot | Subject is small in frame. Environment dominates. Establishing territory. |

- Each shot size maps to an approximate subject-to-frame ratio. The system uses this ratio, combined with the subject's bounding volume, to compute camera distance.
- Shot sizes are not pixel-precise. A medium shot that frames head-to-waist with a little more or less room is still a correct medium shot. The system targets the center of each framing range.
- When the subject is not a person (e.g., "close-up of the clock"), the system adapts framing proportionally -- the object fills roughly the same proportion of frame as the human-referenced definition implies.

``` python
  # shot size with human subject
  .if description is "medium shot" with a character as subject >>
      <== character is framed approximately head to waist
      <== some breathing room exists above the head and at frame edges
      !== character's feet visible
      !== only the character's face visible

  # shot size with non-human subject
  .if description is "close-up of the clock" >>
      <== the clock fills most of the frame, similar to how a face fills a CU
      <== some of the surrounding wall or surface is visible at frame edges

  # abbreviations recognized
  .if description is "MCU of Actor_01" >>
      <== system interprets MCU as medium close-up
      <== framing matches medium close-up definition

  # full names recognized
  .if description is "extreme long shot of the valley" >>
      <== system interprets "extreme long shot" identically to "ELS"
```

### Common Aliases

Cinematographers do not always use the textbook terms. The system recognizes common alternatives:

- "Cowboy shot" = MLS
- "Full shot" = LS
- "Head shot" = CU
- "Choker" = tight CU (between CU and ECU)
- "Two-shot" = MS or MLS framing two subjects
- "Over-the-shoulder" / "OTS" = MS or MCU from behind one subject's shoulder, facing another
- "Single" = any shot size framing one subject
- "Insert" = ECU or CU of an object (not a person)
- "Establishing shot" = ELS or LS with emphasis on environment

``` python
  # alias resolution
  .if description is "cowboy shot of the sheriff" >>
      <== system interprets as medium long shot
      <== character framed head to knees

  # over-the-shoulder requires two subjects
  .if description is "OTS from Actor_01 to Actor_02" >>
      <== camera positions behind Actor_01's shoulder
      <== Actor_02 is the primary subject in frame
      <== Actor_01's shoulder and/or head are visible at frame edge

  .if description is "OTS" with only one character in scene >>
      <== system reports that over-the-shoulder requires two subjects
      <== camera does not reposition
```

### Camera Angles

The system recognizes vertical camera angle descriptions. Each maps to a camera height and tilt relative to the subject.

| Term | Meaning |
|---|---|
| Eye level | Camera at the subject's eye height. Neutral, default. |
| Low angle | Camera below the subject's eye line, tilting up. Subject appears powerful, imposing. |
| High angle | Camera above the subject's eye line, tilting down. Subject appears diminished, vulnerable. |
| Bird's eye | Camera directly or nearly directly above, looking straight down. |
| Worm's eye | Camera at ground level or below, looking straight up. Extreme version of low angle. |
| Dutch angle / Dutch tilt | Camera is level in height but rolled on its forward axis. Frame is tilted. Unease, disorientation. Supports variable roll: "slight dutch" (small roll), "dutch" (moderate roll), "extreme dutch" (large roll). |
| Slightly low / slightly high | Subtle versions of low/high angle. Camera offset from eye level but not dramatically. |

``` python
  # low angle
  .if description is "low angle medium shot of the villain" >>
      <== camera is positioned below the character's eye line
      <== camera tilts upward to keep the character in frame
      <== character appears to loom above the viewer

  # bird's eye
  .if description is "bird's eye of the table" >>
      <== camera is positioned directly above the table
      <== camera points straight down (or nearly so)
      <== the table is visible from above

  # dutch angle applies roll, not height
  .if description is "dutch angle close-up" >>
      <== camera applies a roll rotation (tilting the frame)
      <== camera height is unchanged from what the shot size implies
      <== roll amount is moderate (10-25 degrees) unless description specifies "extreme"

  # dutch angle variable roll amounts
  .if description is "slight dutch" >>
      <== camera applies a small roll (approximately 5-10 degrees)
      <== the tilt is noticeable but subtle

  .if description is "dutch angle" with no intensity modifier >>
      <== camera applies a moderate roll (approximately 10-25 degrees)

  .if description is "extreme dutch" >>
      <== camera applies a large roll (approximately 25-45 degrees)
      <== the frame is dramatically tilted

  # "slightly" modifier reduces intensity
  .if description is "slightly low angle" >>
      <== camera is below eye level but only modestly
      <== the low angle effect is subtle, not dramatic
```

### Spatial Positioning

Descriptions often specify where the camera sits relative to the subject in the horizontal plane. Directional references are relative to camera: "actor on the left" means camera-left (the actor appears on the left side of the frame).

| Term | Meaning |
|---|---|
| Camera left | Camera positioned to the left of the subject (from the camera's perspective) |
| Camera right | Camera positioned to the right of the subject |
| Frontal / head-on | Camera faces the subject directly from the front |
| Profile | Camera is at 90 degrees to the subject -- sees them from the side |
| Three-quarter | Camera is between frontal and profile, approximately 45 degrees |
| From behind / reverse | Camera is behind the subject |

``` python
  # camera left
  .if description is "MCU, camera left" with subject facing the camera >>
      <== camera is offset to the left of center
      <== subject appears slightly right of center in frame
      <== this is a standard interview framing -- subject looking across frame to camera right

  # profile
  .if description is "medium shot, profile" >>
      <== camera is at 90 degrees to the subject
      <== subject is seen from the side

  # directional references are camera-relative
  .if description is "CU of the actor on the left" with multiple characters in scene >>
      <== "left" is interpreted from the camera's perspective
      <== the character who appears on the left side of the current frame is selected as subject
      !== system interprets "left" as stage-left or world-space left

  # ambiguous multi-character descriptions frame the group
  .if description is "medium shot" with multiple characters and no specified subject >>
      <== system frames all characters as a group
      <== camera pulls back far enough to include all characters in the frame
      !== system picks one character arbitrarily
```

### Movement Types (Recognition Only)

The system recognizes movement vocabulary for future use but does not execute movements in this milestone. These terms are parsed, acknowledged, and stored as metadata.

| Term | Meaning |
|---|---|
| Dolly in / dolly out | Camera translates toward / away from subject |
| Track left / track right | Camera translates laterally |
| Crane up / crane down | Camera rises / descends vertically |
| Pan left / pan right | Camera rotates horizontally |
| Tilt up / tilt down | Camera rotates vertically |
| Zoom in / zoom out | Focal length changes (not camera movement) |
| Steadicam | Smooth following movement |
| Handheld | Shaky, documentary-style movement |

``` python
  # movement recognized but not executed
  .if description is "medium shot, slow dolly in toward the actor" >>
      <== system positions camera at the starting framing (medium shot of the actor)
      <== system acknowledges the dolly instruction in feedback
      <== camera does not actually dolly
      <== movement metadata is stored with the shot for potential future use
```

### Compound Descriptions

Real descriptions combine multiple terms. The system must handle composition.

``` python
  # full compound description
  .if description is "medium close-up of the detective, slightly low angle, camera left, profile" >>
      <== shot size: MCU (head to mid-chest)
      <== angle: slightly below eye level
      <== horizontal position: left of subject, at roughly 90 degrees (profile + camera left)
      <== focal length: appropriate for MCU (likely 50-85mm range)

  # all parameters interact
  .if description is "extreme long shot, bird's eye" >>
      <== shot size: ELS (environment dominates)
      <== angle: directly above, looking down
      <== focal length: wide (18-35mm range)
      <== camera positioned high above the scene

  # partial description fills in defaults
  .if description is "close-up" with one character in scene >>
      <== shot size: CU
      <== angle: defaults to eye level
      <== horizontal position: defaults to current camera angle relative to subject
      <== subject: the single character
```

### Vocabulary Limitations

The system's vocabulary is finite. It will not understand every possible description.

``` python
  .if >> description uses non-standard terminology ("a Kubrick shot," "that thing Spielberg does")
      <== system does not recognize the reference
      <== feedback indicates the term was not understood
      !== system guesses based on pop-culture associations

  .if >> description uses non-English language
      <== system does not interpret non-English input in the initial release
      <== feedback indicates that only English is supported

  .if >> description is highly subjective ("make it feel lonely," "something beautiful")
      <== system cannot translate emotional intent into camera parameters
      <== feedback indicates that the description did not contain actionable shot language
```

## Open Questions

These are decisions that may need resolution during development or after user testing:

1. **Confidence thresholds**: At what confidence level should the system refuse to act versus attempt its best guess? Refusing too often makes the feature feel broken. Guessing too often produces wrong results.

2. ~~**Multi-shot descriptions**: Should "OTS from Alice to Bob, then reverse" create two shots in the sequencer, or is this milestone strictly single-shot?~~ **Resolved**: Compound descriptions create multiple shots in the sequencer (see 4.1.1 Compound Descriptions).

3. **Learning from corrections**: If the director manually adjusts every AI-positioned shot by moving the camera 6 inches to the right, should the system learn this preference? This is a future consideration but affects data capture decisions now.

4. **Undo integration**: A shot-from-text action should be undoable as a single undo step (returning the camera to its pre-description position). This depends on 2.1 (Undo/Redo).

5. **Two-shot and OTS framing**: These are among the most common shot types in dialogue scenes but require understanding the spatial relationship between two characters. The spec describes what these should produce but the subject resolution may not be reliable enough for these compound framings.
