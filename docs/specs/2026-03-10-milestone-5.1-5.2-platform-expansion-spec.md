# Milestones 5.1 & 5.2: Platform Expansion — Specification

**Date**: 2026-03-10
**Milestones**: 5.1 (Web), 5.2 (iPad)
**Status**: Draft
**Project**: 5 — Platform expansion

---

- ## 5. Platform expansion (Project)

  Fram3d is a desktop application first. These milestones port it to two additional platforms — the browser and iPad — to remove distribution barriers and enable on-set use. Neither platform adds new features. Everything specified here already exists in the desktop app. The challenge is making it work correctly under each platform's constraints.

  The distribution argument for web is overwhelming: share a link, open a project. No download, no install, no App Store review. Anyone with a browser sees exactly what the director sees. The argument for iPad is physical: a director walks a location, holds up the device, frames the shot through the screen. Touch is the natural input for that workflow.

  These are the last milestones in the roadmap (sequencing principle #6). The desktop app must be solid — every feature from Projects 1 through 4 — before platform porting begins. A web or iPad build that ships before the desktop app is mature will inherit every desktop bug and add platform-specific ones on top.

  *Blocked by: Projects 1, 2, 3, and 4 — every prior milestone*

  ---

  - ### 5.1. Web (Milestone)

    *Fram3d in the browser. Removes the install barrier — share a link, open a project. The same tool, the same project files, accessed from any computer with a browser.*

    The browser is not a neutral container. It has opinions about scroll, about keyboard shortcuts, about focus, about memory. The viewport lives inside a page that can itself scroll. Cmd+S means "save the webpage." Ctrl+R means "reload." Tab switches focus to the address bar. The mouse wheel scrolls the page unless you stop it. WebGL has a memory ceiling that desktop OpenGL does not. Every one of these conflicts must be resolved or the app feels broken — not different, broken.

    Web also introduces a persistence problem the desktop app does not have. There is no local filesystem (not reliably). Projects must live in the cloud. This means accounts, authentication, storage, and sharing URLs. Save/load is no longer a file dialog — it is a network operation that can fail, take time, and require authentication.

    Mobile browsers are explicitly out of scope. The web build targets desktop browsers (Chrome, Firefox, Safari, Edge) with a mouse and keyboard. Touch-screen laptops use mouse input, not touch gestures. A user who opens the link on a phone sees a message directing them to a desktop browser or the iPad app.

    *Blocked by: Projects 1, 2, 3, and 4*

    - ##### 5.1.1. WebGL build (Feature)

      ***The full Fram3d application running in the browser via WebGL — camera system, timeline, scene interaction, overlays — validated for acceptable performance under browser memory and rendering constraints.***

      **Functional requirements:**
      - The web build renders and runs the complete Fram3d feature set: virtual camera, overlays, scene management, shot sequencer, keyframe animation, input system, undo/redo, asset library, export, lighting, characters, multi-camera, hierarchy, and AI features
      - Target browsers: Chrome (latest 2 versions), Firefox (latest 2 versions), Safari (latest 2 versions), Edge (latest 2 versions)
      - The application occupies the full browser viewport — no surrounding page chrome, no scroll
      - The loading experience shows a progress indicator with meaningful stages (not a blank screen or a frozen percentage)
      - If the browser or device does not meet minimum requirements, the user sees an explicit message explaining what is needed — not a blank canvas or a crash
      - Scene complexity limits may differ from desktop — the web build must define and communicate its limits (maximum object count, maximum texture memory) to the user when approached
      - Frame rate target: 30fps minimum for scenes within stated complexity limits

      **Expected behavior:**
      ``` python
        # core loop works
        .if the user opens the Fram3d URL in a supported browser >>
            <== the application loads and displays the 3D viewport
            <== the camera responds to mouse and keyboard input
            <== the timeline and shot sequencer are visible and interactive
            <== overlays (aspect ratio masks, frame guides, HUD) render correctly
            !== the page shows a blank canvas or WebGL error

        # performance under load
        .if the scene contains objects within the stated complexity limits >>
            <== frame rate remains at or above 30fps
            <== camera movements feel responsive (no perceptible input lag)
            <== timeline scrubbing updates the viewport without stutter
        ||> .if the scene approaches complexity limits >>
            <== the user is warned before the scene becomes unresponsive
            !== the browser tab crashes without explanation

        # unsupported browser
        .if the user opens the URL in an unsupported or outdated browser >>
            <== a message states the minimum browser requirements
            !== the application attempts to load and fails silently

        # mobile browser
        .if the user opens the URL on a phone or tablet browser >>
            <== a message directs them to use a desktop browser or the iPad app
            !== the full application attempts to load on a phone screen
      ```

      **Memory and stability:**
      ``` python
        # browser memory ceiling
        .if the user works in a session for an extended period (1+ hours) >>
            <== memory usage remains stable — no unbounded growth
            !== the tab crashes due to memory accumulation

        # tab backgrounding
        .if the user switches to another browser tab >>
            <== the application pauses rendering to reduce resource usage
        ||> .if the user returns to the Fram3d tab >>
            <== the application resumes from exactly where it was
            <== no state is lost
            !== the viewport is black or requires a reload

        # browser refresh
        .if the user refreshes the page (F5 or Cmd+R) >>
            <== the application reloads from scratch
            <== if the project had unsaved changes, the cloud auto-save recovery flow activates (see 5.1.2)
            !== unsaved work is silently lost without warning
      ```

    - ##### 5.1.2. Cloud persistence (Feature)

      ***Save and load projects to cloud storage. Share a project URL that opens directly to a specific project. Replace the desktop file dialog with cloud-native persistence.***

      Cloud persistence replaces the desktop save/load model (2.2) for the web platform. The operations are the same — save, save as, open, recent projects, auto-save, dirty tracking — but the storage backend and sharing model are different.

      **Functional requirements:**
      - Users must have an account to save projects — anonymous/guest users can view shared projects but not save
      - Authentication via email or third-party provider (Google, Apple)
      - **Save** writes the project to the user's cloud storage, associated with their account
      - **Save As** creates a new cloud project with a new name (the original remains unchanged)
      - **Open** shows the user's saved projects in a list — not a file picker
      - **Recent Projects** shows the last 10 accessed projects, same as desktop (2.2.2)
      - Every saved project has a shareable URL
      - Opening a shared URL loads the project in read-only mode for users who are not the owner
      - The owner can grant edit access to specific users by email
      - The project file format is identical to desktop (2.2.1) — a project saved on desktop can be uploaded to web, and a web project can be downloaded as a file
      - Auto-save writes to cloud storage on the same interval as desktop (every 2 minutes while dirty)
      - Auto-save is resilient to network interruption — if a save fails due to connectivity, it retries when the connection is restored
      - The dirty state indicator (asterisk in title) works identically to desktop (2.2.3)
      - A "Download Project" option exports the cloud project as a local file
      - An "Upload Project" option imports a local desktop project file into cloud storage

      **Expected behavior:**
      ``` python
        # saving a project
        .if the user saves a new project >>
            <== a dialog prompts for a project name
            <== the project is saved to cloud storage
            <== the title bar shows the project name without an asterisk
            <== the project appears in the user's project list

        # sharing a project
        .if the user copies the project URL
        .if another user opens that URL >>
            <== the project loads in read-only mode
            <== the viewer can navigate the scene, play back shots, and scrub the timeline
            !== the viewer can modify, save, or export the project

        # shared project with edit access
        .if the owner grants edit access to another user
        .if that user opens the project URL >>
            <== the project loads in edit mode
            <== the user can modify and save (save creates their own copy, unless collaborator permissions evolve in a future milestone)

        # offline / network interruption during save
        .if the user saves the project
        .if the network connection is lost mid-save >>
            <== the user is informed that the save failed due to connectivity
            <== the project remains dirty (asterisk persists)
            <== no partial or corrupted data is written to cloud storage
        ||> .if the network connection is restored >>
            <== auto-save resumes on the next interval
            !== the user must manually retry

        # auto-save resilience
        .if auto-save triggers while the network is unavailable >>
            <== the auto-save is queued
            <== the application continues functioning normally
        ||> .if the network is restored >>
            <== the queued auto-save executes
            !== queued saves accumulate unboundedly

        # desktop-to-web project transfer
        .if the user uploads a .fram3d file saved on desktop >>
            <== the project loads in the web app with full fidelity
            <== the project is saved to cloud storage under the user's account
            !== any data is lost in the transfer

        # web-to-desktop project transfer
        .if the user downloads a cloud project >>
            <== a .fram3d file is downloaded via the browser
            <== the file opens in the desktop app with full fidelity

        # unauthenticated user
        .if a user without an account opens a shared project URL >>
            <== the project loads in read-only mode
            <== a prompt offers sign-up/sign-in if they attempt to save or modify
            !== the project fails to load
      ```

      **Error cases:**
      ``` python
        # deleted or revoked share link
        .if a shared URL points to a project that has been deleted >>
            <== the user sees a message that the project no longer exists
            !== the application crashes or shows a blank viewport

        # storage quota exceeded
        .if the user's cloud storage quota is full
        .if the user attempts to save >>
            <== the user is informed their storage is full
            <== the user is offered the option to download the project as a local file
            !== the save fails silently

        # concurrent access
        .if two users with edit access open the same project simultaneously >>
            <== each user works on their own copy (no real-time collaboration in this milestone)
            <== the last save wins — no merge conflict resolution
            <== both users are warned that another user has the project open
      ```

    - ##### 5.1.3. Web-specific input (Feature)

      ***Adapt the desktop mouse and keyboard control scheme (1.6) for browser constraints — preventing scroll hijacking, resolving keyboard shortcut conflicts with the browser, and handling focus loss gracefully.***

      The desktop input spec (1.6) defines every mouse and keyboard interaction. The web build must reproduce that exact behavior, but the browser sits between the application and the hardware. The browser has its own opinions about what Cmd+S, Ctrl+R, scroll, and Tab do. This feature defines how every conflict is resolved.

      **Functional requirements:**
      - All desktop mouse controls (1.6.1) work identically in the web build
      - All desktop keyboard shortcuts (1.6.2) work identically in the web build
      - Browser default behaviors are suppressed for every key combination that Fram3d uses
      - The application prevents the page from scrolling — the browser scroll event is captured and routed to the camera zoom / dolly / crane system as specified in 1.6.1
      - The following browser shortcuts are intercepted and remapped to Fram3d actions:

      | Browser shortcut | Browser default | Fram3d action |
      |-----------------|-----------------|---------------|
      | Cmd+S | Save webpage | Save project (2.2.2 / 5.1.2) |
      | Cmd+Shift+S | Save As webpage | Save As project |
      | Cmd+Z | Undo (browser) | Undo (2.1) |
      | Cmd+Shift+Z | Redo (browser) | Redo (2.1) |
      | Cmd+O | Open file (browser) | Open project |
      | Cmd+N | New window (browser) | New project |
      | Ctrl+R | Reload page | Reset camera (1.6.2) — the user is never surprised by a page reload |
      | F5 | Reload page | Suppressed — no action |
      | Cmd+R | Reload page (macOS) | Reset camera (1.6.2) |
      | Backspace / Delete | Browser back | Delete keyframe (1.6.2) |
      | Space | Scroll page down | Toggle playback (1.6.2) |

      - The right-click context menu is suppressed over the viewport — right-click is reserved for future Fram3d use
      - The browser's text selection behavior is suppressed — dragging in the viewport does not select page elements
      - Copy (Cmd+C) and Paste (Cmd+V) are available for text fields only — they do not interfere with the C/V keyframe shortcuts when no text field has focus
      - Tab does not move focus to the browser address bar while the application has focus — Tab is reserved for future Fram3d use

      **Expected behavior:**
      ``` python
        # scroll is captured, not passed to browser
        .if the user scrolls over the Fram3d viewport >>
            <== camera zoom activates (per 1.6.1)
            !== the browser page scrolls
            !== the viewport moves within the page

        # Cmd+S saves the project, not the page
        .if the user presses Cmd+S >>
            <== Fram3d save activates (per 2.2.2 / 5.1.2)
            !== the browser "Save Page" dialog appears

        # Ctrl+R resets camera, not page
        .if the user presses Ctrl+R >>
            <== camera resets to default position (per 1.6.2)
            !== the browser reloads the page
            !== the user loses their work

        # Space plays back, does not scroll
        .if the user presses Space
        .if no text field has focus >>
            <== playback toggles (per 1.6.2)
            !== the page scrolls down

        # Backspace deletes keyframe, does not navigate back
        .if a keyframe is selected
        .if the user presses Backspace >>
            <== the keyframe is deleted (per 1.6.2)
            !== the browser navigates to the previous page

        # right-click context menu suppressed
        .if the user right-clicks on the viewport >>
            !== the browser context menu appears

        # text field focus resolves Cmd+C/V vs C/V
        .if a text field has focus (e.g., shot name)
        .if the user presses Cmd+C >>
            <== browser copy activates (copies selected text)
            !== a camera keyframe is added
        ||> .if no text field has focus
        .if the user presses C >>
            <== a camera keyframe is added (per 1.6.2)
            !== browser copy activates

        # Tab stays in the application
        .if the user presses Tab while interacting with the viewport >>
            !== browser focus moves to the address bar
            !== keyboard shortcuts stop working
      ```

      **Focus management:**
      ``` python
        # application loses focus
        .if the user clicks outside the Fram3d canvas (e.g., the browser address bar) >>
            <== keyboard shortcuts stop working (expected — the browser has focus)
        ||> .if the user clicks back on the Fram3d canvas >>
            <== keyboard shortcuts resume immediately
            <== the application state is unchanged
            !== a stuck modifier key causes unexpected behavior

        # modifier key stuck across focus change
        .if the user holds Cmd and clicks the address bar (focus leaves the app)
        .if the user releases Cmd while the address bar has focus
        .if the user clicks back on the Fram3d canvas >>
            <== modifier state is cleared — Cmd is not treated as held
            !== the next scroll is interpreted as dolly (Cmd+scroll) when no modifier is held

        # browser alert or dialog steals focus
        .if a browser permission dialog or alert appears >>
            <== the application pauses input handling
        ||> .if the dialog is dismissed >>
            <== input handling resumes
            <== no inputs are buffered or replayed from while the dialog was active

        # window/tab switching
        .if the user switches to another application or tab and returns >>
            <== the application resumes exactly where it was
            <== modifier key state is cleared on return
            !== a drag operation is stuck in progress
      ```

  - ### 5.2. iPad (Milestone)

    *Fram3d on iPad. Touch-first interaction for on-set and location scouting. The director holds the iPad, frames the shot, walks the set.*

    iPad is a fundamentally different input surface. There is no mouse cursor, no hover state, no right-click, no modifier keys, and no scroll wheel. Every interaction from the desktop spec (1.6) must be reimagined for fingers and gestures. The good news: iOS gesture conventions are well-established and intuitive. Pinch to zoom, two-finger drag to pan — people already know these. The challenge is mapping Fram3d's richer interaction model (six distinct scroll+modifier combinations, two distinct drag+modifier combinations, a full keyboard shortcut table) onto a gesture vocabulary that has far fewer dimensions.

    Apple Pencil adds precision that fingers lack. A finger tap selects; a Pencil tap selects with sub-millimeter accuracy. A finger drag on a gizmo is coarse; a Pencil drag on a gizmo handle is precise. The spec must distinguish finger input from Pencil input where the interaction quality differs.

    The iPad also has hard constraints the desktop does not: no filesystem dialog, memory limits on older devices, the on-screen keyboard covering half the viewport when editing text, multitasking modes that resize the app, and no persistent hover state for UI tooltips.

    *Blocked by: Projects 1, 2, 3, and 4*

    ---

    - ##### 5.2.1. Touch controls (Feature)

      ***Remap every camera and scene interaction from mouse+keyboard to multi-touch gestures. Pinch to zoom, two-finger pan, three-finger orbit. Toolbar buttons replace keyboard shortcuts.***

      **Gesture reference — Viewport:**

      | Gesture | Action | Desktop equivalent |
      |---------|--------|--------------------|
      | One-finger tap | Select object (raycast from touch point) | Left click |
      | One-finger tap on empty space | Deselect | Left click on empty space |
      | One-finger drag on gizmo handle | Manipulate gizmo (translate/rotate/scale) | Left click-drag on gizmo |
      | Two-finger pinch | Zoom (focal length) | Scroll Y (no modifier) |
      | Two-finger drag | Pan + tilt (camera) | Ctrl-drag |
      | Three-finger drag | Orbit | Alt-drag |
      | Two-finger vertical swipe (fast) | Dolly in/out | Cmd + Scroll Y |
      | Long press | Context menu (tool mode, object options) | Right-click (future) / keyboard shortcuts |

      **Gesture reference — Timeline:**

      | Gesture | Action | Desktop equivalent |
      |---------|--------|--------------------|
      | One-finger tap on timeline | Scrub to tapped time | Left click on timeline |
      | One-finger tap on keyframe | Select keyframe | Left click on keyframe |
      | One-finger drag on keyframe | Reposition keyframe | Left drag on keyframe |
      | One-finger tap on shot thumbnail | Select shot | Left click on shot thumbnail |
      | One-finger long press on shot thumbnail | Reorder (drag mode activates) | Drag-and-drop on thumbnail |

      **Toolbar:**

      Desktop keyboard shortcuts are replaced by an on-screen toolbar for common actions:

      | Toolbar button | Action | Desktop shortcut |
      |---------------|--------|-----------------|
      | Select tool | Select mode (no gizmo) | Q |
      | Move tool | Translate mode | W |
      | Rotate tool | Rotate mode | E |
      | Scale tool | Scale mode | R |
      | Play / Pause | Toggle playback | Space |
      | Add camera keyframe | Add camera keyframe at current time | C |
      | Add object keyframe | Add object keyframe (if object selected) | V |
      | Focus | Focus on selected object | F |
      | Undo | Undo | Cmd+Z |
      | Redo | Redo | Cmd+Shift+Z |

      Additional controls accessible via menus or secondary toolbars: camera body/lens cycling, aspect ratio selection, overlay toggles, focal length presets, save/load.

      **Functional requirements:**
      - Every interaction available on desktop is available on iPad — no features are missing, only the input method differs
      - Gestures are resolved by finger count — the system distinguishes one-finger, two-finger, and three-finger gestures without ambiguity
      - Gesture recognition uses finger count at gesture start — adding a finger mid-gesture does not change the gesture type
      - One-finger gestures in the viewport are reserved for selection and gizmo manipulation — one-finger drag on empty viewport space does nothing (prevents accidental camera movement while trying to select)
      - Two-finger gestures in the viewport are always camera operations
      - Three-finger gestures in the viewport are always orbit
      - Gestures over the timeline panel affect only the timeline — they do not move the camera
      - The toolbar is always visible and does not overlap the viewport or timeline in a way that blocks interaction
      - The toolbar adapts to the current state: the active tool is highlighted, the add-object-keyframe button is dimmed when no object is selected

      **Expected behavior:**
      ``` python
        # pinch to zoom
        .if the user performs a pinch-in gesture on the viewport >>
            <== focal length decreases (zoom out)
            !== the camera translates
        ||> .if the user performs a pinch-out gesture >>
            <== focal length increases (zoom in)

        # two-finger drag pans and tilts
        .if the user drags two fingers horizontally on the viewport >>
            <== the camera pans (yaws) in the drag direction
            !== the camera translates sideways
        ||> .if the user drags two fingers vertically >>
            <== the camera tilts (pitches) in the drag direction
        ||> .if the user drags two fingers diagonally >>
            <== the camera pans and tilts simultaneously

        # three-finger orbit
        .if an object is selected
        .if the user drags three fingers on the viewport >>
            <== the camera orbits around the selected object
            <== the selected object remains at approximately the same screen position
        ||> .if no object is selected
        .if the user drags three fingers >>
            <== the camera orbits around the world origin

        # tap to select
        .if the user taps a scene object with one finger >>
            <== the object is selected
            <== the selection highlight appears
            <== the active gizmo appears (if a tool mode other than select is active)
        ||> .if the user taps empty space >>
            <== the current selection is cleared

        # gizmo manipulation
        .if an object is selected with a visible gizmo
        .if the user drags a gizmo handle with one finger >>
            <== the object transforms along the gizmo axis
            !== camera movement occurs
            !== the object is deselected

        # finger count determines gesture
        .if the user places two fingers and begins dragging >>
            <== the gesture is resolved as pan/tilt
        ||> .if the user adds a third finger during the drag >>
            <== the gesture continues as pan/tilt (gesture type locked at start)
            !== the gesture switches to orbit

        # long press for context menu
        .if the user long-presses on the viewport >>
            <== a context menu appears with tool modes and common actions
            !== the object under the finger is dragged

        # one-finger drag on empty space does nothing
        .if the user drags one finger on empty viewport space (not on a gizmo, not on an object) >>
            <== nothing happens
            !== the camera moves
            !== any object is selected

        # toolbar interaction
        .if the user taps the translate tool button >>
            <== translate mode activates
            <== the translate button shows as active
            <== the previous tool button is deactivated
        ||> .if the user taps the play button >>
            <== playback starts
            <== the play button icon changes to a pause icon

        # timeline gestures
        .if the user taps the timeline area >>
            <== the playhead moves to the tapped time
            !== the camera moves
        ||> .if the user drags a keyframe on the timeline >>
            <== the keyframe repositions (snaps to 0.1s)
            !== the camera moves
      ```

      **Interaction during playback:**
      ``` python
        # touch during playback
        .if playback is active
        .if the user taps an object >>
            <== playback pauses
            <== the object is selected

        .if playback is active
        .if the user performs a two-finger gesture (camera) >>
            <== playback pauses
            <== the camera movement takes effect
      ```

      **Edge cases:**
      ``` python
        # gesture near viewport/timeline boundary
        .if the user begins a two-finger drag near the boundary between viewport and timeline >>
            <== the gesture is assigned to whichever region contains the centroid of the initial touch points
            !== the gesture affects both regions

        # accidental palm touch
        .if the user touches the screen with their palm while holding the iPad >>
            <== palm rejection prevents unintended input
            !== objects are accidentally selected or deselected
            !== the camera moves unexpectedly

        # external keyboard connected
        .if a hardware keyboard is connected to the iPad >>
            <== desktop keyboard shortcuts (1.6.2) are active
            <== toolbar buttons remain available as an alternative
            <== both input methods work simultaneously without conflict
      ```

    - ##### 5.2.2. Apple Pencil support (Feature)

      ***Precision selection and gizmo manipulation with Apple Pencil. Pencil input is distinguished from finger input — fingers handle camera gestures, Pencil handles scene interaction.***

      The Apple Pencil provides two advantages over finger input: sub-millimeter positional accuracy and pressure sensitivity. Both matter for gizmo manipulation, where selecting a specific axis handle and dragging with fine control is the difference between "roughly there" and "exactly there."

      **Functional requirements:**
      - The application distinguishes Pencil input from finger input at all times
      - Pencil tap selects objects (same as one-finger tap, but with higher precision)
      - Pencil drag on a gizmo handle manipulates the gizmo with fine control
      - Pencil pressure modulates gizmo manipulation speed: light pressure = slow/precise, firm pressure = fast/coarse
      - Pencil does not trigger camera gestures — camera control is always fingers
      - Pencil hover (on supported iPads) shows a preview cursor indicating what would be selected or which gizmo handle would be grabbed
      - Pencil double-tap (on supported Pencils) toggles between the last two active tool modes
      - When the Pencil is active for gizmo manipulation, fingers remain available for simultaneous camera gestures (e.g., user orbits with fingers while dragging a gizmo with Pencil)

      **Expected behavior:**
      ``` python
        # Pencil selects with precision
        .if two objects are close together on screen
        .if the user taps with the Pencil on one of them >>
            <== the tapped object is selected (precise hit test)
            !== the nearby object is selected due to touch target inflation

        # Pencil gizmo manipulation
        .if an object is selected with a visible gizmo
        .if the user drags a gizmo handle with the Pencil >>
            <== the object transforms along the gizmo axis
            <== movement is smooth and precise (no touch-target snapping)

        # pressure sensitivity
        .if the user drags a gizmo handle with light Pencil pressure >>
            <== the object moves slowly (fine adjustment mode)
        ||> .if the user increases pressure during the drag >>
            <== the object moves faster (coarse adjustment mode)
            <== the transition between speeds is smooth, not stepped

        # Pencil does not orbit or zoom
        .if the user drags the Pencil across the viewport (not on a gizmo) >>
            !== the camera pans, tilts, or orbits
            <== the drag is treated as a selection drag (future marquee select) or ignored

        # simultaneous Pencil + finger input
        .if the user is dragging a gizmo handle with the Pencil
        .if the user simultaneously pinches with two fingers >>
            <== the gizmo drag continues
            <== the camera zooms simultaneously
            <== both inputs are processed independently

        # Pencil hover preview
        .if the iPad supports Pencil hover
        .if the user hovers the Pencil over a gizmo handle >>
            <== the handle highlights (same visual as desktop mouse hover)
        ||> .if the user hovers over a scene object >>
            <== the object shows the hover highlight
        ||> .if the user hovers over empty space >>
            <== no highlight is shown

        # Pencil double-tap tool toggle
        .if the user is in translate mode
        .if the user was previously in rotate mode
        .if the user double-taps the Pencil barrel >>
            <== the tool mode switches to rotate
        ||> .if the user double-taps again >>
            <== the tool mode switches back to translate

        # Pencil on timeline
        .if the user taps the timeline with the Pencil >>
            <== the playhead moves to the tapped time (same precision as viewport)
        ||> .if the user drags a keyframe with the Pencil >>
            <== the keyframe repositions with high precision
            <== pressure sensitivity applies: light = fine, firm = coarse
      ```

      **Edge cases:**
      ``` python
        # Pencil battery dead
        .if the Apple Pencil loses charge during use >>
            <== all Pencil-specific interactions fall back to finger equivalents
            <== the user is not blocked from any functionality
            !== the application enters an error state

        # Pencil not paired
        .if no Apple Pencil is connected >>
            <== all interactions work via finger input only
            <== Pencil-specific features (hover, pressure sensitivity) are simply absent
            !== the application shows errors or missing-feature warnings

        # Pencil double-tap when only one tool used
        .if the user has only used one tool mode this session
        .if the user double-taps the Pencil barrel >>
            <== no tool change occurs (there is no "previous" tool to switch to)
      ```
